# 🧮 估值建模（model-builder）— 筆記

推薦 🧮 估值建模(model-builder)。理由是它跟你剛做的市場研究天然接續，而且仍然完全不必碰內部系統。

## 為什麼是 model-builder

劇情上剛好接得起來：

```text
🔭 市場研究 幫你「找出值得關注的名字」(Vertiv、nVent…)
      │
      ▼
🧮 估值建模 幫你「把其中一家從零建出估值模型」
```

技術上仍然無痛：

- ✅ 同一個家族(研究與建模)、同一批公開連接器(capiq/daloopa)
- ✅ 不需要自建任何 .mcp.json
- ✅ 沒金鑰也能跑——你可以「手動餵假設數字」，輸出是一份獨立 Excel 模型
- ✅ 產出不對外、不過帳——建完先停給你審(風險中)

對照另一個候選 📊 財報追蹤(earnings-reviewer)：它預設你已經有一份在追蹤的模型要更新，而且更依賴資料源餵實際業績/逐字稿——起步門檻比 model-builder 高一點。所以先 model-builder，再 earnings-reviewer。

## 安裝(跟上次一模一樣的套路)

```bash
/plugin install model-builder@fsi-local
```

financial-analysis 上次已經裝過，這次只要裝主角這一支。

## 第一句指令(從你剛做的散熱研究延伸)

> 使用者：幫 Vertiv（VRT）從零建一份現金流折現（DCF）估值模型，列出關鍵假設，並做一張 WACC × 永續成長率的敏感度表

## 這支「新增要注意」的三件事(跟市場研究不同的地方)

① 它會「寫 Excel 檔」——市場研究主要產文字，這支產試算表

② 它有「建完先停」的關卡——建好模型會停下等你核准，才往下做敏感度

③ 它有獨立「稽核(auditor)」——會自我檢查：每個結果都能追回原始輸入、計算格不打死數字(藍輸入／黑公式／綠連結的色碼慣例)

## 驗收檢查表(沿用上次的習慣)

- ☐ 計算格是不是「公式」，而不是被寫死的數字？
- ☐ 沒接金鑰時，抓不到的輸入有沒有標 [ASSUMPTION]＋註明來源/假設？（而不是憑空填一個漂亮數字）
- ☐ 有沒有在「建完模型」這一步停下來等你審，而不是一路衝到底？
- ☐ 敏感度表有沒有出來（DCF 通常是 WACC × 成長率）？
- ☐ 每個輸出能不能往回追到輸入？

🔴 一樣的破口：沒金鑰時，歷史財務/市場預期這些輸入會掛 [ASSUMPTION] 或要你手動給——這是正常行為。之後接 FactSet/CapIQ 金鑰就會自動填實。
