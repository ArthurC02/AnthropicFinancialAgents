一、這套方案是什麼：10 個金融具名代理（Named Agent）

先講結論：這個 repo 給你的不是一堆零散工具，而是 10 位「已經受過職前訓練的數位新進員工」。 技術上的正式名稱是「具名代理」（named agent）——每支 agent 都以它負責的那一條工作流程命名，工作內容取材自投資銀行（investment banking）、證券研究（equity research）、私募股權（private equity）、財富管理（wealth management）這四個垂直領域的日常業務。每支 agent 會從頭到尾自己跑完一件事：讀資料、做分析、產出草稿，然後停下來等人簽核——這種「機器做工、人守關卡」的做法，行話叫人工把關（HITL，human-in-the-loop）。

另外一個重點：每支 agent plugin（外掛套件）都是自包含（self-contained）的。 意思是它把自己會用到的 skills（技能包——可以想成塞給新進員工的 SOP 手冊）一起打包好了。 裝一個 agent，就等於拿到它需要的全部東西，不必再東拼西湊。

分類 Agent（資料夾 slug） 用途 風險
客戶經營與顧問（前台） meeting-prep-agent（會前準備） 開會前彙整客戶關係、持倉、市場動態，產出會前資料包 🟢 低
pitch-agent（投行提案） 做三種估值分析——comps（可比公司分析，comparable companies analysis：拿同業的行情來比價）、precedents（先例交易分析，precedent transactions：參考過去類似交易的成交價）、LBO（槓桿收購，leveraged buyout：評估「借錢買下這家公司」的划算程度）——產出估值模型與掛上品牌版型的提案簡報。所謂估值（valuation），就是估算一家公司值多少錢 🟡 中
研究與建模（前台） market-researcher（市場研究） 給定產業或主題 → 產出概況、競爭格局、同業比較、標的清單 🟡 中
earnings-reviewer（財報追蹤） 讀法說會與財報 → 更新模型 → 產出觀點筆記草稿 🟡 中
model-builder（估值建模） 從零建立 DCF（現金流折現法，discounted cash flow：把公司未來會賺的現金換算成今天的價值）、LBO、三大報表與 comps 模型，最後落地成 Excel 檔 🟡 中
基金行政與財務（後台） valuation-reviewer（估值複核） 接收 GP（基金管理人，general partner）送來的估值包、跑估值模板，備妥給 LP（出資人，limited partner）的報告 🔴 高
gl-reconciler（總帳對帳） 找出總帳（GL，general ledger，公司整體帳務的總紀錄）與明細帳（subledger，一筆一筆交易的細項紀錄）之間的差異、追出根本原因、送人簽核 🔴 高
month-end-closer（月底結帳） 估列應計費用、餘額結轉、差異說明 🔴 高
statement-auditor（對帳單稽核） 對帳單寄給客戶之前，逐欄核對基金淨值（NAV，net asset value：基金資產減掉負債之後，每一單位值多少錢） 🔴 高
開戶審查（中台） kyc-screener（開戶審查） 解析開戶文件、跑規則引擎，把缺漏標出來交給合規人員 🔴 高

這張表的分組（客戶經營與顧問、研究與建模、基金行政與財務、開戶審查）直接對應 README.md「Agents」表格的四個功能列。 🟢🟡🔴 的風險分層則來自 docs/AgentSummary.md「一、十個助理一覽」。分層原則很簡單：越靠近「錢」跟「法規」，風險越高、把關越嚴。 前台產出的是內部草稿，有人 review 就能攔錯；後台碰的是帳目與對出資人的交代，錯不得。

名稱不等於範圍「pitch-agent」「gl-reconciler」這些名字對應的是這套 repo 預設的工作流程，不是「投行部門」或「對帳部門」的全部工作。docs/AgentSummary.md 開頭就講明：這是把「贏得客戶 → 開戶 → 基金日常營運 → 對出資人交代」這條 pipeline 串起來的一套班底，而不是銀行全業務的覆蓋圖。

一句話總結：這套 repo 給你 10 支各自負責一條完整工作流程的具名代理，而且越靠近錢和法規的那幾支，風險分級越高、人工把關越嚴。

二、一個來源、兩種輸出：Plugin vs CMA

先講結論：同一支 agent，可以「坐在你旁邊上班」，也可以「在雲端值夜班」——用的是同一份大腦。 README.md 第一段講的原則是這本書的技術主軸：「Everything here is available two ways from one source」。 也就是同一份 agents/<slug>.md 系統提示詞（system prompt——agent 的「職位說明書」，定義它是誰、能做什麼、規矩是什麼）、同一批 skills，由你決定它跑在哪裡。 managed-agent-cookbooks/README.md 說得更直白：「Same agent, same skills — pick your surface.」

誰在盯著它跑：人。分析師在 Cowork 介面裡點開 agent、看它一步步做事，隨時可以插話、中止、要求改方向。就像新人坐在你隔壁，你隨時能探頭看一眼。

怎麼裝：Cowork 裡走 Settings → Plugins → Add plugin，貼 repo URL 或上傳 zip；Claude Code 則是先加 marketplace（插件市集），再逐一 claude plugin install。

下面這段指令做的事，就是「把市集加進來，然後裝三個套件」。非工程讀者可以直接跳過，不影響後面閱讀。

Claude Code — 安裝核心 skills 與 agent
claude plugin marketplace add anthropics/financial-services

claude plugin install financial-analysis@claude-for-financial-services
claude plugin install gl-reconciler@claude-for-financial-services
claude plugin install market-researcher@claude-for-financial-services

裝完之後，agent 會出現在 Cowork 的 dispatch 清單裡；skills 會在相關時機自動觸發；slash command（斜線指令，例如 /comps、/dcf、/earnings）則可以在 session（工作階段）裡直接叫用。

誰在盯著它跑：沒有人即時盯。CMA（Claude Managed Agents）是無人值守（headless）模式——由你自己的後端系統在特定事件發生時透過 API（應用程式介面——系統之間互相呼叫的標準窗口）呼叫它。

怎麼跑：cookbook（部署食譜）目錄裡的 managed-agent-cookbooks/<slug>/agent.yaml 是部署清單（deploy manifest），直接引用同一份 plugin 目錄下的系統提示詞與 skills。 執行 scripts/deploy-managed-agent.sh <slug>，腳本會把 skills 上傳、建好 leaf-worker subagents（子代理——主代理底下分工的小幫手）， 再用真正的 POST /v1/agents 欄位把 orchestrator（主代理，負責調度 leaf worker 的那一支）建起來。

下面兩行是實際的部署指令：第一行設定 API 金鑰，第二行部署 gl-reconciler 這支 agent。

部署一支 CMA
export ANTHROPIC_API_KEY=sk-ant-...
scripts/deploy-managed-agent.sh gl-reconciler
關鍵事實：運算在誰家？CMA 的實際運算跑在 Anthropic 雲端（POST /v1/agents，beta header managed-agents-2026-04-01）。你的銀行 AWS 帳號裡放的是周邊系統——呼叫端後端、MCP（Model Context Protocol，模型情境協定——你可以把它想成 agent 接外部系統的標準插座，用來連總帳、CRM、行情庫這類資料源）servers、編排器（scripts/orchestrate.py）與資料源。放在 AWS 的是這些配套，不是 agent 本身的運算。

那份 agent.yaml 部署清單長什麼樣？主線讀者只要知道：它就是一張「到職單」， 寫明這支 agent 用哪份職位說明書（系統提示詞）、帶哪些 SOP（skills）、底下編制哪些小幫手（leaf workers）。想逐欄看懂的人，展開下面的進階區塊。

managed-agent-cookbooks/README.md 給了 YAML（一種給人讀也給機器讀的設定檔格式）欄位對照： system: {file: ..., append: "..."} 會被解析成完整系統提示詞字串；skills: [{from_plugin: ...}] 會把整個 plugin 目錄下的 skills 上傳成 {type: custom, skill_id: ...}；callable_agents（可呼叫代理清單）裡每一筆 {manifest: ./subagents/x.yaml} 則建出對應的 leaf worker。實際看一支範例：

managed-agent-cookbooks/pitch-agent/agent.yaml（節錄）
name: pitch-agent
model: claude-opus-4-7

system:
  file: ../../plugins/agent-plugins/pitch-agent/agents/pitch-agent.md
  append: "You are running headless. Produce files in ./out/; do not assume an open Office document."

mcp_servers:
  - { type: url, name: capiq,   url: "${CAPIQ_MCP_URL}" }
  - { type: url, name: daloopa, url: "${DALOOPA_MCP_URL}" }

skills:
  - { from_plugin: ../../plugins/agent-plugins/pitch-agent }

callable_agents:
  - { manifest: ./subagents/researcher.yaml }
  - { manifest: ./subagents/modeler.yaml }
  - { manifest: ./subagents/deck-writer.yaml }   # only leaf with Write

有一件事必須留在主線講：上面清單裡的 mcp_servers 用了 ${CAPIQ_MCP_URL} 這種環境變數（environment variable——部署時才填入的可替換參數）佔位。CMA 版本上線時只換這裡的 URL——從 mock（假資料的模擬服務，供開發測試用）換成真實供應商——不改 agent 裡的 server 名稱。因為 MCP 是以「名字」解析的：agent 只認得「capiq」這個名字，名字背後接誰，由部署清單決定。

一句話總結：Plugin 與 CMA 用的是同一份系統提示詞和同一批 skills，差別只在跑的地方和誰在盯著——Plugin 有人看著跑，CMA 由後端自動呼叫、運算在 Anthropic 雲端。

三、為什麼銀行場景選 CMA

先講結論：銀行的核心整合場景沒有人會替每一筆案件手動點開視窗，所以要選能被系統呼叫的 CMA。 Plugin 模式的前提是「有個人坐在 Cowork 前面」，這對投行分析師、研究員很自然。但授信審核排程、夜間批次對帳、事件觸發的爭議調查，都是機器對機器的流程。 銀行工程師要的是一個能被自己的工作流引擎（Temporal、Airflow，或既有的事件匯流排）當成服務接進去的東西。這正是 CMA 的定位：managed-agent-cookbooks/README.md 說 CMA 是「your platform team deploys behind your own workflow engine」——由你的平台團隊部署在自家工作流引擎後面。

三個具體理由：

1. 後端驅動，不需要人守著

事件發生（一筆撥款申請進來、一批對帳資料落地）就觸發 API 呼叫；agent 跑完寫結果、發 webhook（回呼通知——跑完主動回報你的系統，而不是要你一直去問）。全程不需要分析師開視窗盯著。

2. 能掛進既有工作流引擎

CMA 是一個 API 端點，不是一個桌面 App。它可以被排程系統、事件匯流排、既有的核心銀行系統當成「其中一個步驟」呼叫，銜接你原本就有的簽核、記錄、重試機制。

3. 權限靠架構鎖死，不是靠信任

callable_agents 目前只支援一層委派：orchestrator 能呼叫 leaf worker，leaf worker 不能再往下叫。而且每支 agent 的 leaf worker 裡只有一個有 Write 權限（managed-agent-cookbooks/README.md 表格用粗體標出，例如 gl-reconciler 是 resolver）。誰能寫、誰只能讀，白紙黑字寫在部署清單裡，不是靠提示詞裡一句「請小心」。

具名代理之間也不會互相呼叫。可以想成部門之間不准直接塞單子，一律走公文交換中心。一支 agent 需要另一支接手時，它在輸出裡吐出一筆 交棒訊號（handoff）——欄位名是 handoff_request——由 scripts/orchestrate.py（或銀行自己的事件匯流排）把它路由成一個新的 steering event（觸發 agent 上工的事件訊息），送到目標 session。用一個流程圖看實際發生順序：

1Steering event
後端呼叫 POST /v1/agents 觸發 orchestrator（例如「Reconcile GL vs subledger, trade date D」）
→
2Leaf workers（一層）
orchestrator 呼叫 reader → critic → resolver，僅 resolver 有 Write
→
3handoff_request
需要另一支 agent 接手時，輸出裡吐出 handoff 訊號，自己不直接呼叫別的 agent
→
4orchestrate.py 路由
allowlist 目標＋驗證 payload，轉成新 steering event 送給下一支 agent

這支參考腳本在路由 handoff 時做兩件事：第一，硬性比對 allowlist（白名單——只有名單上的目標 agent 收得到交棒，不在名單上的一律拒絕）；第二，對交棒帶的 payload 做 schema（資料格式規格書）驗證，欄位不合格式就擋下來。也就是說，即使某支 agent 的輸出被污染、想把工作丟給不該收的對象，路由層也不會放行。完整的路由設計在第 8 章展開。

一句話總結：銀行選 CMA，是因為它能被既有工作流引擎當成服務呼叫，而且「誰能寫、誰能叫誰」是寫死在部署清單裡的架構限制，不是口頭承諾。

四、10 個 Agent 對映銀行業務：落點與缺口

先講結論：這 10 支對投行、研究、基金行政、財富管理的覆蓋很好；對零售與商業銀行的核心業務，目前是空白。 誠實地看：這 10 支 agent 是從投資銀行、證券研究、私募股權、財富管理四個垂直（README.md「Vertical Plugins」表）長出來的，服務的是投行前台、研究部、PE 基金行政、財富管理顧問——不是零售或商業銀行的核心放貸與清算業務。對映到銀行實際要顧的業務，落點大致是：

銀行業務 對映的現有 agent 吻合程度
私人銀行／財富管理客戶會議 meeting-prep-agent 高——場景幾乎直接可用
法人金融部研究、產業覆蓋 market-researcher、earnings-reviewer、model-builder 高
投資銀行部門提案作業 pitch-agent 高
基金／代操業務的每日對帳、月結、季末估值 gl-reconciler、month-end-closer、valuation-reviewer、statement-auditor 中高——邏輯相通，但銀行總帳科目表跟基金不同，需客製
私人銀行客戶開戶 KYC（認識你的客戶，know your customer——開戶前確認客戶身分與資金來源的審查程序）／AML（洗錢防制，anti-money laundering） kyc-screener 中——規則引擎框架可用，但零售銀行開戶規則、名單來源跟基金 LP 審查不同

而下面這些銀行天天要處理的業務，這 10 支 agent 裡沒有一支直接對應：

  • 授信審核（企業/個人貸款的信用評估與核貸建議）
  • 信用卡爭議調查（持卡人異議、疑似盜刷的調查與裁決建議）
  • 房貸審件（貸款文件審查、估價比對、核貸條件試算）
  • 逾期催收（帳齡分級、催收話術與升級路徑建議）
  • 分行臨櫃客戶服務（存款帳戶異動、爭議交易查詢）

這個缺口不是 repo 的疏漏，而是它從一開始就鎖定的範圍。但這也正是本書後半段要處理的：第 4 章「客製」教你怎麼把 gl-reconciler、kyc-screener 這類已有的 agent 骨架（reader → critic/rules-engine → 唯一有 Write 的 leaf）換上銀行自己的科目表、規則、名單來源；第 5 章「打造新 Agent」則教你當現有 10 支都對不上——例如要做一支信用卡爭議調查 agent——怎麼從零仿照這套「髒 reader 隔離、schema 鎖 enum、驗證方式對抗任務性質」的設計模式，長出屬於銀行自己的第 11 支 agent。

一句話總結：這 10 支 agent 覆蓋投行、研究、基金行政與財富管理場景；授信、卡務、房貸這些零售核心業務沒有現成的，要靠第 4、5 章的客製與新建自己補上。

六、本章重點

本章重點這套 repo 是 10 支「具名代理」，同一份系統提示詞與 skills 能以 Plugin（人監督，裝在 Cowork/Claude Code）或 CMA（無人值守，走 POST /v1/agents，跑在 Anthropic 雲端，銀行 AWS 帳號放的是周邊系統）兩種形式部署。銀行後端整合場景通常選 CMA，因為它能掛進既有工作流引擎，且權限（一層委派、單一 leaf 有 Write）是架構鎖死而非人工承諾。10 支 agent 對映的是投行/研究/PE/財富管理/基金行政，授信、信用卡爭議、房貸審件等零售/商業銀行核心業務目前沒有對應 agent——這正是第 4、5 章要補的缺口。
Q：關於 Claude Managed Agents（CMA），下列敘述何者正確？
「部署在 AWS」講的是周邊系統（後端、MCP servers、orchestrate.py、資料源），CMA 本身的 agent 運算是呼叫 Anthropic 的 POST /v1/agents（beta header managed-agents-2026-04-01）在 Anthropic 雲端執行。Plugin 與 CMA 引用的是同一份 agents/<slug>.md 系統提示詞與同一批 skills，差別只在跑的地方跟誰在盯著。
