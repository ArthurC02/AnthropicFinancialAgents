Chapter 8編排與跨 Agent 協作

一個 具名代理（Named Agent）可以把工作派給自己的 葉子工人（leaf worker），但它不能直接呼叫另一個具名代理。所以「valuation-reviewer 發現一筆有問題的 portco（portfolio company，基金投資的被投資公司），要通知 gl-reconciler」這種跨 agent 協作，靠的不是 API 呼叫。靠的是一份純文字宣告——handoff_request（交棒請求：請求把工作交接給另一個 agent）——再加上銀行自己的編排器。本章假設你的 agent 已經部署上線（第 7 章），接著拆解三件事：兩層委派的邊界在哪裡、handoff_request 怎麼被驗證與路由、以及把參考腳本 orchestrate.py 換成 Temporal / Airflow / Step Functions 時，哪些安全職責絕對不能丟。

🔑 開始前，你需要先懂
  • orchestrator 與 leaf worker 的分工（第 0 章第 3 節）——主代理派工給自己的工人，工人不能再往下派。
  • 第 7 章：AWS 部署——agent 都部署好、能被觸發執行了，才談得上「多個 agent 之間怎麼接力」。

8.1 兩層編排：agent 內委派 vs agent 間協作

先講結論：這個 repo 裡的「編排」（orchestration）分成兩層，範圍完全不同。很多人第一次看會搞混，所以先用一個銀行的比喻把邊界畫清楚。

想像一家分行。第一層像是「部門內派工」：授信部主管把案件拆給自己組裡的三位承辦，主管盯進度、承辦各做各的，全程在同一個部門裡完成。第二層像是「部門之間的公文往返」：授信部要通知法遵部，不能自己抱著卷宗衝進別人辦公室，一律走公文交換中心，由收發室核對收件單位、格式無誤後才轉送。兩層的規則、負責人、風險都不一樣。

對回技術上的精確定義。第一層是 agent 內部的 callable_agents：一個具名代理（orchestrator，主代理）呼叫自己的葉子工人。這層委派完全在 Anthropic 雲端執行，是 POST /v1/agents 這個 research preview（研究預覽階段，尚未正式開放「GA，General Availability」）API 端點原生支援的能力，而且只支援一層——工人不能再往下叫工人。

第二層是 agent 之間的協作。比如 valuation-reviewer 發現一筆有問題的 portco 後，要讓 gl-reconciler 知道；或 kyc-screener 篩出高風險客戶後，要讓人工覆核佇列知道。這一層 Anthropic 完全不管。靠的是銀行自己的編排器（scripts/orchestrate.py 或你既有的工作流引擎），把一個 agent 的輸出路由成另一個 agent 的新 steering event（引導事件——丟進一個執行中 session 的一則新文字指令；session 是 agent 一次連續的執行工作階段）。

兩層的差異可以用「誰負責檢查」來記。第一層的安全邊界，是 agent.yaml 裡宣告好的 tools／連結器（見第 9 章的三層隔離）——Anthropic 端執行，銀行不用插手驗證。第二層完全相反：一個 agent 的文字輸出，本質上是不可信輸入的下游產物（見 8.3 節），所以路由前必須由銀行的程式碼把關。Anthropic 不會替你做這件事。

與第 9 章的關係三層工具隔離（reader / orchestrator / writer）解決的是「一個 agent 內部，未信任文件的資料不能直接觸發寫入」；本章解決的是「一個 agent 的結論，怎麼安全地變成另一個 agent 的輸入」。兩者都假設輸出可能被污染，只是污染的來源和下游動作不同。

一句話總結：部門內派工（callable_agents）由 Anthropic 雲端原生支援、只有一層；部門間公文（handoff_request）Anthropic 不管，收發把關全是銀行編排器的責任。

8.2 Steering event：一次執行怎麼被觸發

Steering event 是銀行後端丟給一個已部署 CMA（Claude Managed Agent，Claude 託管代理）session 的一段純文字指令，對應到 client.beta.agents.sessions.steer(agent_id=..., input=...) 這支 API。你可以把它想成「丟一張工作單進某個承辦的收件匣」。每個 agent 的 steering-examples.json 定義了它預期收到的事件長什麼樣子——注意，這不是說明文件，是實際會被丟進 session 的字串。幾個真實例子：

Agent steering event（來自 steering-examples.json） 場景
pitch-agent Build pitch book: target CRWD, acquirer PANW, thesis: platform consolidation in security 指定標的與收購方的併購簡報
kyc-screener Screen onboarding packet PKT-2026-00318 新戶開戶審查
kyc-screener Re-screen UBOs only for packet PKT-2026-00318 after updated ownership chart 補件後的追加事件（同一 packet，範圍縮小）
month-end-closer Close entity US-OPCO for period 2026-04 標準月結
statement-auditor Tie out statement batch BATCH-2026Q1-GIII against fund Growth-III NAV pack 季度 LP（有限合夥人，基金出資人）對帳單全批次核對

注意每份 steering-examples.json 都混了「起手事件」和「追加事件」。同一個 session 可以被 steer 多次：kyc-screener 收到補件後只重篩 UBO（實質受益人），不用整包重來；month-end-closer 收到晚到的分錄（late JEs）後，只重寫 variance commentary（差異說明備註）。這對應到銀行後端一個很實際的問題——不是每次觸發都要開全新 session，追加類事件應該 steer 進既有 session。

那執行結果怎麼回來？CMA 的 session 用 SSE（Server-Sent Events）串流回傳——像逐字聽寫，agent 每產出一小段文字，就推一小段給訂閱端。銀行後端訂閱這條串流，逐段檢查文字內容。這也是為什麼「跨 agent 協作」必須另外設計：一個 agent 的輸出只是文字增量，銀行的程式碼要自己從中辨認出「這是不是一個要求交接給別的 agent 的信號」。

參考腳本 scripts/orchestrate.py 訂閱來源 session 的方式如下：

scripts/orchestrate.py（節錄，SSE 事件迴圈）
with client.beta.agents.sessions.stream(session_id=source_session_id) as stream:
    for event in stream:
        if event.type != "message_delta" or not getattr(event, "text", None):
            continue
        handoff = extract_handoff(event.text)
        ...

逐行看：stream(...) 用 context manager 開一條對來源 session 的 SSE 連線；迴圈裡只關心 message_delta 型別、而且帶有文字的事件（其他型別如工具呼叫進度直接 continue 略過）；每一段文字增量都丟給 extract_handoff()（見 8.3 節）掃一次，看看裡面有沒有交接宣告。要注意的是：因為文字是「增量」進來的，正式實作通常還要處理 JSON 被切在兩個 delta 之間的緩衝問題——參考腳本示範的是形狀，不是完整品。

一句話總結：steering event 就是丟進 session 收件匣的一張純文字工作單，同一個 session 可以收很多張；結果經由 SSE 逐段流回來，要靠銀行自己的程式碼從文字裡認出交接信號。

8.3 handoff_request 機制與威脅模型

先講結論：具名代理之間永不互相呼叫。當一個 agent 判斷工作該交給另一個 agent 時，它只能在自己的文字輸出裡「宣告」一個 handoff_request，格式大致是 {"type": "handoff_request", "target_agent": "...", "payload": {...}} 這樣一段 JSON。

回到公文交換中心的比喻，把這件事講透。部門之間不直接搬檔案：授信部想通知法遵部，只能寫一張公文，投進交換中心。這張公文本身沒有任何執行力——它只是紙。真正決定「送不送、送給誰」的是收發室，收發室做兩件事：第一，核對收件單位是否在名單上（不在名單，公文直接作廢）；第二，核對公文格式是否合規（欄位缺漏、夾帶不明附件，一樣作廢）。handoff_request 就是那張公文，scripts/orchestrate.py 就是收發室：allowlist（白名單——只放行清單內的目標）對應「收件單位核對」，schema（資料格式規格書）驗證對應「格式核對」。兩關都過，收發室才會呼叫 steer() 把工作單真正投遞給目標 agent。

為什麼要這麼防？因為這裡有一個真實的攻擊面，本質上就是提示注入（prompt injection）的下游版本。腳本開頭的註解把威脅模型寫得很白：

orchestrate.py 的威脅模型（原文摘要）

攻擊面：handoff request 出現在 orchestrator 的文字輸出裡，而這個輸出是「未信任文件讀取者（untrusted-document readers）」的下游產物。一個能控制被處理文件內容的攻擊者，可以在文件裡嵌入一段一模一樣格式的 handoff_request JSON——如果 orchestrator 把文件內容原文照抄進輸出，這段偽造的交接請求就會被這支腳本解析、路由出去。

兩道緩解：(a) target_agent 對照部署好的 slug 做 hard allowlist——不在名單裡一律丟棄；(b) payload 進 jsonschema.validate 做 schema 驗證，格式不對就丟棄，不會被拿去 steer 下一個 session。

作者自己承認的局限：這只是參考腳本，正式環境建議改用「專屬 tool call」或「模型無法靠引用文件文字偽造的 typed SSE event」來發出交接請求——換句話說，比對純文字更可靠的作法，是讓交接訊號走一條攻擊者摸不到的通道。

兩道驗證在程式碼裡長這樣：

scripts/orchestrate.py（節錄，extract_handoff）
ALLOWED_TARGETS = {
    "pitch-agent", "market-researcher", "earnings-reviewer", "meeting-prep-agent",
    "model-builder", "gl-reconciler", "kyc-screener",
    "valuation-reviewer", "month-end-closer", "statement-auditor",
}

HANDOFF_PAYLOAD_SCHEMA = {
    "type": "object",
    "additionalProperties": False,
    "required": ["event"],
    "properties": {
        "event": {"type": "string", "maxLength": 2000},
        "context_ref": {"type": "string", "maxLength": 256,
                        "pattern": r"^[A-Za-z0-9 ._/:#-]+$"},
    },
}

def extract_handoff(text: str) -> dict | None:
    m = HANDOFF_RE.search(text)
    if not m:
        return None
    obj = json.loads(m.group(0))
    target = obj.get("target_agent")
    payload = obj.get("payload")
    if target not in ALLOWED_TARGETS:
        return None
    jsonschema.validate(instance=payload, schema=HANDOFF_PAYLOAD_SCHEMA)
    return {"target_agent": target, "payload": payload}

幾個值得注意的設計：ALLOWED_TARGETS 是寫死在腳本裡的常數集合，收錄的是「已部署的 agent slug」，不是從任何輸出動態產生——這就是「hard」allowlist 的意思。schema 那邊，additionalProperties: False 擋掉任何多塞的欄位；event 是唯一必填、上限 2000 字元；選填的 context_ref 除了 256 字元上限，還用正規表達式把字元集鎖在英數與少數符號，讓攻擊者很難在裡面夾帶注入片段。流程上：正規表達式先撈出候選 JSON、解析、查 target、驗 payload——任一步不過就回 None，整段交接請求等於不存在。

把整個機制串起來看：agent 完成工作 → 在輸出裡宣告 handoff_request → orchestrate.py 攔截、做 allowlist + schema 兩道驗證 → 通過就呼叫 client.beta.agents.sessions.steer()，把 payload.event 當成新的 steering event 丟給目標 agent 的 session。三個真實案例（來自各 agent README 的 Handoff 小節）：

  • valuation-reviewer 發現有問題的 portco，emit handoff_request 給 gl-reconciler
  • gl-reconciler 驗證完 break（對帳差異，未平帳項目）後，emit handoff_request 給 month-end-closer，把結果折進月結 commentary
  • earnings-reviewer / pitch-agent / market-researcher 在需要重建模型時，都 emit handoff_request 給同一個 model-builder

三個來源交接給同一個目標，正是「target 必須做 allowlist、而不是信任來源」這件事的具體理由。model-builder 的 README 也直接寫明：「when invoked from earnings-reviewer or pitch-agent, the calling agent's handoff_request is routed here by scripts/orchestrate.py」——它不知道也不需要知道呼叫者是誰，反正 orchestrate.py 已經把關過。就像收發室蓋章放行的公文，收件部門不必再回頭查發文者。

一句話總結：handoff_request 只是一張沒有執行力的公文；收發室（orchestrate.py）核對收件名單（allowlist）與公文格式（schema），兩關都過才真正投遞，這兩道關卡是防偽造交接的安全底線。

8.4 換成銀行既有的編排引擎

orchestrate.py 檔頭第一句話就是「REFERENCE ONLY — replace with your firm's workflow engine (Temporal, Airflow, Guidewire event bus)」。這支腳本示範的是「迴圈的形狀」，不是要你拿去上線。換句話說：收發室的規則要照搬，收發室的建築可以換成你家的。三個常見引擎（Temporal、Airflow、Step Functions + EventBridge）各自怎麼對應，細節收在下面的進階區塊；不管選哪一個，三個安全職責必須原封不動搬過去。

用一個長駐 Workflow 訂閱來源 session 的 SSE（可以包成一個 Activity），在 Workflow 內做 extract_handoff 那段解析。allowlist 用 Workflow 的常數表，schema 驗證可以直接搬 jsonschema（Python worker）或換成 Temporal 支援語言的等價套件。驗證通過後，下一步 Activity 呼叫 steer() 對目標 session 送新事件；Temporal 原生的重試（retry policy）與工作流歷史（history）天生就適合當審計軌跡——這正好省了 8.6 節要自己刻的重試/冪等（idempotency，同一事件重複執行、結果不變、不會重複做事）邏輯。

來源 agent 完成後，由一個 sensor 或事件觸發的 DAG 去讀輸出（session 結束後的最終文字，而不是即時 SSE）。解析、allowlist、schema 驗證各自是一個 task；驗證失敗直接讓 DAG task 失敗並落到 Airflow 的失敗通知，不要吞掉錯誤。Airflow 的 task retry + XCom 可以留一份「這次 handoff 驗證結果」的記錄當審計軌跡，但要注意 Airflow 排程天生是輪詢式的，即時性不如 Temporal/SSE 訂閱式。

來源 agent 的後端呼叫方在收到 session 完成後，把輸出 PutEvents 到 EventBridge，帶上 detail-type 如 agent.handoff_candidate。一個 Step Functions 狀態機接手：Lambda 做 allowlist + schema 驗證（同一段邏輯，換個執行環境），驗證通過才進入下一個狀態呼叫目標 agent 的 steer API；驗證失敗的分支直接進 SQS 死信佇列（DLQ）而不是靜默丟棄。EventBridge 的事件本身自帶時間戳與來源，天生適合接稽核系統。

換引擎時不能丟的三件事
  1. Hard allowlist：target_agent 永遠對照「已部署的 slug 清單」，清單來自你自己的部署紀錄，不是來自 agent 輸出本身。
  2. Payload schema 驗證：驗證失敗要「安靜地丟棄」，不能因為想保留彈性而放寬 additionalProperties 或拿掉 maxLength。
  3. 審計軌跡：每一次 handoff 的來源 session、目標 agent、驗證結果（通過/拒絕）、時間戳都要留痕——這是第 9、10 章合規審查會要看的東西，不是編排引擎的附加功能。

一句話總結：orchestrate.py 只是示範用的收發室模型，換成 Temporal / Airflow / Step Functions 都行，但 allowlist、schema 驗證、審計軌跡三件事一件都不能少。

8.5 銀行情境串接範例

下面兩個情境。第一個是 repo 目前實際文件化的交接鏈路（gl-reconciler → month-end-closer，寫在 gl-reconciler 的 cookbook 說明檔 managed-agent-cookbooks/gl-reconciler/README.md 裡）——gl-reconciler 是負責總帳（GL，General Ledger）對帳的 agent。第二個是把同一套機制延伸到 KYC（認識你的客戶）審查 agent kyc-screener 的高風險升級——但它的升級對象是「人工覆核佇列」而不是另一個具名代理，值得特別拆解兩者的差異。

情境 A：gl-reconciler 驗證完 break → month-end-closer 折進月結

1gl-reconciler 完成一次對帳
trade date + asset class 清單，reader 只讀不信任的對帳單，resolver 驗證後產出 exception report（差異清單報告）
→
2輸出宣告 handoff_request
target_agent: "month-end-closer"，payload 帶上已驗證的 break 摘要
→
3orchestrate.py 驗證
allowlist 命中、schema 通過 → 呼叫 steer()
→
4month-end-closer 收到新 steering event
把驗證過的 break 折進該實體當期的 variance commentary

這是文件裡實際寫出來的鏈路。兩邊都是具名代理，走的完全是 8.3 節那條 handoff_request 路徑——公文從對帳部門發出，經收發室兩道核對，投遞到月結部門的收件匣。

情境 B：kyc-screener 發現高風險 → 人工覆核佇列（不是 handoff_request）

1kyc-screener 篩完一個開戶包
doc-reader 抽取欄位、rules-engine 跑規則／PEP（政治敏感人物，Politically Exposed Person）／制裁名單比對
→
2escalator（唯一有 Write 權限的 葉子工人）產出檔案
./out/escalation-<packet>.xlsx，附建議風險等級
→
3銀行案件系統接手，不是 orchestrate.py
因為目標不是「另一個部署好的 agent slug」，走不進 ALLOWED_TARGETS，本來就不該套用 handoff_request 這條路徑
→
4合規官在案件系統裡做最終決定
README 寫明：「this agent recommends a risk rating; the compliance officer decides」

情境 B 刻意示範一個容易搞混的地方：orchestrate.py 的 ALLOWED_TARGETS 只收錄「已部署的 agent slug」。人工覆核佇列不是 agent，塞不進這個機制，也不該硬塞——升級到人工佇列該用銀行既有的案件管理系統，不必也不能透過 handoff_request 表達。用比喻說：要通知的對象根本不是另一個部門，而是「主管親簽」，那就不是公文交換的事。

如果銀行想讓 month-end-closer 完結後接著觸發 statement-auditor（比如某支基金的實體月結完成後，順帶核對當季 LP 對帳單），機制上完全可行——statement-auditor 本來就已經在 ALLOWED_TARGETS 清單裡（見 8.3 節程式碼）。但這條鏈路目前不是 repo 既有文件裡寫出來的預設交接（目前唯一文件化的下游是 gl-reconciler → month-end-closer）。要接的話，銀行只需要兩步：在 month-end-closer 的 system prompt 加上「完結時視情況 emit handoff_request 給 statement-auditor」的指示，並確認 payload 的 event 欄位塞得下要傳的 batch/fund 資訊——不需要動 orchestrate.py 的 allowlist 或 schema。

一句話總結：agent 對 agent 才走 handoff_request；升級給「人」（合規官、覆核佇列）走銀行既有的案件系統，兩條路不要混用。

8.6 失敗處理：重試、冪等、死信佇列

先講結論：跨 agent 協作一旦接上銀行的編排器，失敗處理就是銀行自己的責任。CMA 不會替你重試 steering 失敗的呼叫。幾個必須設計的點：

  • 重試：steer() 呼叫失敗（網路、rate limit〔速率限制〕、目標 session 已結束）要重試，但重試前先確認下一點
  • 冪等（idempotency）：同一個 handoff_request 因為 SSE 重連或重試被解析兩次，不能對目標 agent steer 兩次——用 context_ref（schema 裡已有的欄位）或來源 session_id + target_agent 組一個去重 key，處理過的直接跳過
  • 死信佇列（DLQ）：schema 驗證失敗、target 不在 allowlist、或重試次數用盡的 handoff，不能靜默丟掉——進 DLQ 保留原始文字與判定結果，讓人工事後查得到「有一個交接請求被擋下來，為什麼」
  • 人為介入點（HITL，human-in-the-loop）：DLQ 裡的項目、以及所有「Not guaranteed」類的輸出（風險等級建議、pass/hold 建議）都要有明確的人工審核關卡，不能因為編排器暢通就把最後一道人工簽核也自動化掉

這幾點不是編排引擎的加分項。它們是把 orchestrate.py 那句「REFERENCE ONLY」落地成正式系統時的必要條件——參考腳本本身完全沒有重試、去重、DLQ，只示範了 happy path（一切順利時的路徑）的解析與路由邏輯。

一句話總結：重試、冪等去重、死信佇列、人工簽核關卡，這四件事參考腳本一件都沒做，全部要由銀行在正式編排器裡補上。

本章重點
  • 編排分兩層：callable_agents 是 agent 內部一層委派，在 Anthropic 雲端執行；handoff_request 是 agent 之間協作，靠銀行編排器（orchestrate.py 或 Temporal/Airflow/Step Functions）路由
  • 具名代理 之間永不互相呼叫；handoff_request 只是輸出裡的一段文字宣告（一張公文），有沒有執行力完全取決於編排器（收發室）
  • orchestrate.py 的威脅模型：agent 輸出是未信任文件讀取的下游產物，攻擊者可偽造 handoff_request；緩解靠 hard allowlist + schema 驗證兩道關卡，正式環境建議走專屬 tool call 或 typed SSE event
  • 換成銀行既有引擎時，allowlist、schema 驗證、審計軌跡三件事必須原封不動保留
  • 不是每個「升級」都走 handoff_request——升級給人工佇列（如 kyc-screener 的高風險升級）目標不是 agent slug，本來就不套用這個機制
Q：某個攻擊者在 gl-reconciler 讀取的一份客戶對帳單裡，嵌入了一段偽造成 {"type":"handoff_request","target_agent":"model-builder",...} 的文字，希望誘導 orchestrator 把它原文複述出來、進而被 orchestrate.py 路由出去。下列何者最準確描述 orchestrate.py 對此的緩解方式？
orchestrate.py 檔頭註解明講這個威脅模型：handoff request 出現在「未信任文件讀取者的下游輸出」裡，攻擊者能透過文件內容誘導 orchestrator 複述出偽造的 handoff_request。兩道緩解是 (a) target_agent 對照部署好的 slug 做 hard allowlist，(b) payload 做 jsonschema 驗證，兩者缺一都會被丟棄；作者也承認這只是文字比對，正式環境更可靠的做法是用專屬 tool call 或模型無法透過引用文件偽造的 typed SSE event。
