# 📅 Month-End Closer（月底結帳）

> **用途**：月結作業助理，計算應計與結轉、說明科目差異，最終產出給控制員的月結包。

`跑月底結帳（應計／結轉／差異說明）→ 結帳包給控制員簽核`

| 家族 | 用戶 | 頻率 | 風險 |
|---|---|---|---|
| 後台作業（基金行政） | 控制員 | 每月底 | 🔴 高 |

## 一、Workflow

```
 實體 + 期間 (YYYY-MM)
     │
     ▼
┌──────────┐  ┌────────────┐  ┌────────────┐  ┌────────────┐
│①拉試算表 │─►│②建應計+結轉│─►│③差異說明   │─►│④組裝結帳包 │
│GL總帳    │  │必須對得上  │  │解釋為什麼  │  │poster組裝  │
│          │  │foot check  │  │不瞎掰      │  └─────┬──────┘
└──────────┘  └─────▲──────┘  └────────────┘        │
                    │                                ▼
                內建自我驗證                   控制員簽核 ⚖️
```
> 應計＝已發生未付的費用先估列入帳　|　結轉＝證明「期初＋變動＝期末」對得上　|　刻意不做：日常對帳

**步驟拆解**
1. **拉出試算表** — 依指定公司與月份，調出該月的試算表（全部科目餘額一覽），當作結帳起點。
   - 步驟說明：總帳資料拉取——透過 GL MCP（`internal-gl` 內部總帳，唯讀）指定 entity 與 period（YYYY-MM），抓出該期間全科目餘額作為結帳基礎。
2. **算應計與結轉** — 把這個月該認列卻還沒入帳的費用先估列（應計），並把各科目從月初推算到月底（結轉），確認前後對得上。
   - 步驟說明：應計與結轉建表＋內建驗算——用 `accrual-schedule` 估列應計、`roll-forward` 推算結轉，並以內建 foot check 驗算「期初＋變動－沖回＝期末」必須對得上（自我驗證，無外部 critic）。
3. **說明差異** — 對變動較大的科目，從實際發生的事情解釋為什麼變這麼多，不硬湊數字。
   - 步驟說明：差異說明（flux）——`variance-commentary` 對超過重大性門檻的科目做差異分析，從底層實際活動解釋變動原因，原因不明就標示不瞎掰。
4. **組裝結帳包** — 檢查無誤後，把前面成果整理成一份結帳包，暫存等主管簽核。
   - 步驟說明：結帳包組裝——由 poster 用 `audit-xls`（檢查 Excel）與 `xlsx-author`（產出報告檔）把前面成果整理成結帳包，暫存等控制員簽核（agent 永不過帳）。

## 二、風險與把關

```
 agent ──草擬分錄──► 控制員 ⚖️ 核准後才過帳    ✗ agent 永不過帳

 發票/對帳單(外來) ──► reader(無MCP·無寫) ──► 結構化 ──► 流程
 (不可信)
```
> 護操守：對不上就攤開不硬湊 · 原因不明就標示不瞎掰　|　最後防線：不過帳

## 三、技術架構

**技能鏈**
```
accrual-schedule ──► roll-forward ──► variance-commentary ──► (audit-xls / xlsx-author)
應計表             結轉表(must foot)   差異說明
```
**Skill（共 5 支）**　`accrual-schedule` 應計表 · `roll-forward` 結轉表 · `variance-commentary` 差異說明 · `audit-xls` 檢查 Excel · `xlsx-author` 產出報告檔
**MCP（1 個）**　`internal-gl` 內部總帳（唯讀）

> 工具：讀／搜尋＋總帳MCP（唯讀）　⚠️ 無寫檔·無委派　|　模型：opus-4-7

**外掛 vs CMA（兩種安裝）**
```
 外掛版 (真人盯)         CMA版 (雲端·無頭)
 ┌───────────┐          ┌──────────────────────────┐
 │ 單一agent │          │ 主代理 (不寫)            │
 │ 工具收窄  │          │  ├ ledger-reader 讀帳    │
 └───────────┘          │  ├ rollforward   建結轉  │
                        │  └ poster        唯一可寫 │
                        └──────────────────────────┘
```
> 🆕 驗證特色：**沒有外部 critic**——自我驗證內建在「結轉必須對得上（foot check）」的檢查裡。因為結帳是「建構平衡表」，不是「找誤報」，驗證設計隨任務性質而不同

**跨 agent**　fund-admin 三兄弟 ┄ 對帳（gl-reconciler）╳ 本 agent（月底結帳）╳ 報表稽核

## 五、導入評估

| 面向 | 評估 |
|---|---|
| **導入風險** | 🔴 高 — 月結直接影響財報數字與財務誠信，雖然 agent 只產出結帳包、由控制員（controller）核准後才過帳，但底層應計／結轉／差異一旦失真，會污染對外財報。覆核可攔錯，但風險本質屬合規與財務誠信，不是純品質問題。 |
| **導入成本** | 🔴 高 — `internal-gl` 是空殼，要接公司內部總帳（拉試算表／查分錄／查餘額）；還要客製應計排程、結轉規則、重大性門檻（預設五成分之一）與「一定要說明」科目清單，外加分錄格式與會計科目表（chart of accounts）。整合重、規則多。 |
| **適用單位** | 基金行政（後台作業）、財務結帳／總帳部、控制（control）部門 |
| **單位中角色** | 控制員（下指令＋核准過帳，唯一可過帳者）· 結帳會計（產結帳包＋覆核應計結轉）· 財務主管（簽核結帳包＋核重大差異說明） |

**最有機會成功的場景**
1. **例行月結的應計與結轉打底** — 每月固定要估列應計、推算結轉並驗證「期初＋變動＝期末」對得上，agent 把這段重複勞動自動化，控制員只需覆核，CP（成本效益）值最高。
2. **重大差異的初步說明草稿** — 超過重大性門檻的科目要寫差異說明（flux），agent 從底層活動先擬一版草稿，會計補上業務脈絡後定稿，省下大量人工撈資料時間。
3. **結帳包的組裝與自我檢查** — 把試算表、應計表、結轉表、差異說明組裝成一份結構化結帳包並跑內建 foot check，控制員拿到的是已自我驗算過的版本，簽核更快。
