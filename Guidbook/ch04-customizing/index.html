Chapter 4客製化既有 Agent

大多數銀行第一次碰這個 repo,需求都不是「做一支全新的 agent」,而是「把現成的 gl-reconciler 改成我們家的科目表」或「kyc-screener 要加金管會名單」。這一章講清楚:什麼情況下改劇本就夠、改哪個檔案、改完要跑哪三道檢查,以及一份劇本同時餵給 Cowork 外掛和 Claude Managed Agent(CMA)雲端版是怎麼運作的。

1. 調整就夠,還是該開新 Agent?

先講結論:「調整」改的是同一支 agent 的內容(系統提示、skill、模型、CMA 結構);「開新 agent」是另立一份 agents/<slug>.md 劇本、另一組信任邊界。 判斷不是看改動大小,是看下面三個問題有沒有變:

問題 1:「What you produce」清單裡的東西,還是同一種嗎?

gl-reconciler 產出的永遠是「break list / root-cause trace / 對帳異常報告」;如果你要的是全新一種產出(例如「併購案估值模型」),那不是同一支 agent 的客製,是第 5 章的「新增 Agent」。

→ 是同一種產出,那信任邊界(誰會碰到未受信任文件)有變嗎?

gl-reconciler 現有的 reader 已經在碰 custodian/subledger 這類外部文件;kyc-screener 現有的 doc-reader 已經在碰 onboarding PDF。只要新規則、新資料源不會新增一種「原本不碰、現在要碰」的外部文件類型,信任邊界沒變。

→ 沒變,調整就夠。往下看第 2 節,決定要改哪個檔。

這是本章多數情境:換科目表、換法規名單、換語言、換簽核流程 — 都是同一支 agent 在做同一件事,只是換了輸入資料或門檻。

→ 變了(要碰一種全新類型的外部文件/新工具),重新畫安全分層

通常還不用整支開新 agent — 可以在同一支 agent 裡新增一個 reader 分層(新的 leaf worker,即第 2 章講過的「葉子工人」,一樣 Read/Grep only、無 MCP),沿用第 9 章的「碰髒資料的 worker 不給寫檔權」原則即可,不必離開這一章的範圍。

→ 不是同一種產出,開新 Agent(見第 5 章)

另立 agents/<slug>.md、另一組 tools:、需要的話再開一份 CMA cookbook。不要把新產出硬塞進既有 agent 的 Workflow — 那樣會讓既有的安全分層(哪個 tier 碰髒資料)變得說不清楚。

docs/Customizing.md 把這句話寫成一句記法:「角色/流程/守則」改劇本(兩邊生效);「怎麼做一件事」改 SKILL.md(要 sync);「分工/權限/結構」改 CMA 的 yaml(只 CMA)。下一節就照這三類展開,另外加一層「模型」單獨講,因為它常被誤會成要改程式碼,其實多數時候只是換 prompt 裡的假設。

2. 可調整的四個層面

四個層面對應到四種檔案、兩種影響範圍(只影響 plugin 外掛版,還是 plugin + CMA 雲端版都動到)。先看總表,後面逐一展開。

層面動這個檔影響範圍要注意
① 系統提示plugins/agent-plugins/<slug>/agents/<slug>.mdplugin + CMAcanonical 來源,CMA 的 agent.yaml 用 system.file 指回它
② Skillplugins/vertical-plugins/<vertical>/skills/<name>/SKILL.md(真本)plugin + CMA改完必須 sync-agent-skills.py,否則 check.py 擋
③ 模型視 Layer:SKILL.md(算法)/ agent.md 的 Skills 行(掛哪些)/ prompt(假設數字)plugin + CMA 或 單次多數需求其實是 Layer 3(換假設),在 prompt 講就好
④ CMA 專屬結構managed-agent-cookbooks/<slug>/agent.yaml、subagents/*.yaml、steering-examples.json只 CMAsystem.append、tools 開關、sub-agent 分工

① 系統提示 — 一份劇本兩邊生效

以 gl-reconciler 為例,plugins/agent-plugins/gl-reconciler/agents/gl-reconciler.md 是唯一的 canonical 系統提示,裡面有 ## Workflow、## Guardrails、Skills 清單。CMA 版不會另外寫一份提示 — managed-agent-cookbooks/gl-reconciler/agent.yaml 直接指回它:

managed-agent-cookbooks/gl-reconciler/agent.yaml
system:
  file: ../../plugins/agent-plugins/gl-reconciler/agents/gl-reconciler.md
  append: "You are running headless. Produce files in ./out/; do not assume an open Office document."

所以改 Workflow 的某一步(例如把「人工停點」搬到別的步驟),或改 Guardrails 的某條守則,兩種殼(plugin 外掛、CMA 雲端版)都同時生效,不用兩邊各改一次。CMA 額外用 append 補一句「無頭模式」的行為差異 — 這是 CMA 專屬的微調,見第 ④ 層。

② Skill — 真本→sync→check 鐵律

Skill 的內容(門檻、公式、判斷規則)住在 plugins/vertical-plugins/<vertical>/skills/<name>/SKILL.md,agent 包底下的 plugins/agent-plugins/<slug>/skills/<name>/ 只是「vendored copy(搬運副本)」。scripts/sync-agent-skills.py 的邏輯很直白:掃過所有 vertical 底下的 skill 目錄建索引,對每一個已存在的 agent 包 skill 目錄,shutil.rmtree 整個刪掉再 shutil.copytree 從真本重貼一次。

兩個容易踩的坑,都來自這個機制:

  • 直接改 agent 包裡的副本(不是真本)— check.py 的 4b 步驟會用 filecmp.dircmp 逐檔比對真本與副本,有差異就擋 commit
  • 指望 sync 會「新增」skill 到 agent 包 — 它只更新已存在的目錄;第一次要用的 skill,得先手動把真本整個資料夾複製進 agent 包一次,之後 sync 才會接手維護

標準流程一定是這三步:

1改真本
vertical-plugins/<vertical>/skills/<name>/SKILL.md
→
2sync
python3 scripts/sync-agent-skills.py 把所有已存在的 agent 包副本重新同步
→
3check
python3 scripts/check.py 驗 drift + 參照 + skill 是否真的在 agent 自己的 bundle 裡

check.py 另有一條 4b2:凡是 agents/<slug>.md 用反引號提到的 skill 名稱,必須真的存在於該 agent 自己的 skills/ 資料夾裡 — 這條就是為了讓「劇本掛了但沒 bundle 進去」這種半調子改動在本地就被抓到,不用等雲端部署才炸。

③ 模型 — 先分清楚要改哪一層

docs/Models.md 把「改模型」拆成三層,順序決定你要動多少檔案:

層改什麼動哪裡例子
Layer 1模型內部邏輯/慣例vertical-plugins/.../<model>/SKILL.md 真本 → syncDCF 的 WACC 組成、色碼規則
Layer 2agent 會建哪幾種模型agents/<slug>.md 的 Skills 行 + Workflowmodel-builder 加掛 merger-model
Layer 3單次的數字/假設執行時的 prompt,不改檔「WACC 用 9%、永續成長 2.5%」

Models.md 特別點出兩支「repo 裡已經有原始碼、但沒有任何 agent 預設掛」的模型 — merger-model(併購 accretion/dilution,住 investment-banking vertical)和 unit-economics(SaaS 單位經濟,住 private-equity vertical)。要用不用重寫,照 Layer 2 的步驟掛進去就好。另外要注意「plugin 版有掛、CMA 版沒掛」的落差:pitch-agent 的 comps-analysis/3-statement-model 寫在共用劇本裡,plugin 版會用,但 CMA 的 subagents/modeler.yaml 只掛了 dcf-model/lbo-model;valuation-reviewer 的 portfolio-monitoring 也是同樣狀況,CMA 的 subagents/valuation-runner.yaml 只掛了 returns-analysis。換模型時如果雲端版也要跑,記得補進對應的 subagent yaml,別以為改了 agents/<slug>.md 就兩邊都到位 — 模型是唯一一個「劇本掛了、CMA 工人清單卻要另外補」的例外。

④ CMA 專屬 — agent.yaml 的三個開關

CMA cookbook 有三處是純雲端結構,plugin 外掛完全不涉及:

  • system.append — 在 canonical 系統提示後面補一段雲端專屬行為,像 gl-reconciler 的「headless 模式,產出寫到 ./out/」。
  • tools: 的 agent_toolset_20260401 / mcp_toolset 開關 — 每個工具預設關閉(default_config: { enabled: false }),逐一 configs: 白名單打開。kyc-screener 的 orchestrator 只開 read/grep/glob 三個內建工具 + screening 這個 MCP,其餘一律不給。
  • steering-examples.json — 給雲端 session 的啟動事件範例,像 kyc-screener 的三筆:「Screen onboarding packet PKT-2026-00318」「Periodic refresh: client C-004921」「Re-screen UBOs only」。

要加減一個 sub-agent,改 agent.yaml 的 callable_agents 清單,另外寫一支 subagents/<name>.yaml(記得 callable_agents 只支援一層委派,orchestrator 叫得動 leaf worker,leaf worker 不能再叫下一層)。

3. 實戰:改成台灣銀行情境

用兩個真實存在的 agent 示範「改前(repo 現況)」→「改後(客製示範)」。改後的內容是示範用途,實際科目編碼、法條依據上線前務必請會計/法遵覆核 — 這正是 Customizing.md 提醒的「業務門檻要 sign off」。

範例一:gl-reconciler 加台灣銀行科目表 mapping 規則

真本在 plugins/vertical-plugins/fund-admin/skills/gl-recon/SKILL.md。現況的 Step 3 只把「科目對應錯誤」列成一種抽象的 break 原因,沒有具體科目表:

gl-recon/SKILL.md — Step 3(節錄)
- **Mapping** — security or account mapped to a different GL account
  than expected
gl-recon/SKILL.md — Step 3(加一段「台灣銀行科目表」章節)
- **Mapping** — security or account mapped to a different GL account
  than expected(見下方「台灣銀行科目表」對照表,先查表再判斷是否為 mapping break)

## 台灣銀行科目表對照(示範,實際編碼以本行會計政策為準)

| 科目大類 | 科目代碼區間 | 常見 break 型態 |
|---|---|---|
| 現金及約當現金 | 11xx | timing break(跨行清算日差) |
| 存放央行及拆借同業 | 12xx | FX break(拆款利率日 vs 記帳日) |
| 買匯貼現及放款 | 13xx | mapping break(擔保品科目誤植) |
| 存款及匯款 | 21xx | quantity break(存款利息計提差) |
| 損益類科目 | 5xx/6xx | fee/accrual break |

科目代碼由 GL extract 的 `account` 欄位前兩碼判斷大類;命中「買匯貼現及放款」
且變異金額 < 1,000 元,優先標記為 mapping,不是 data quality。

改完真本後,依序跑 python3 scripts/sync-agent-skills.py(把 plugins/agent-plugins/gl-reconciler/skills/gl-recon/ 這份副本同步)、python3 scripts/check.py(確認 drift 清空、參照仍完整),就能用 /plugin install gl-reconciler@fsi-local 本地跑一次驗證行為。

同樣是改真本 → sync → check 三步。因為 kyc-screener 的 tools: 只開了 read/grep/glob + screening 這個 MCP,加名單來源只是換 kyc-rules 裡「怎麼判斷」的邏輯,不需要動 agent.yaml 的工具開關 — 這也印證了第 2 節的判斷:同一種產出、信任邊界沒變,調整 Skill 就夠,不必碰 CMA 結構。

4. 三道安全閘:改完別急著 push

這裡講的是「改動有沒有守規矩」的閘,跟第 10 章 agent 執行期的能力閘/schema 閘/交棒閘是不同層次的三道閘 — 這三道閘管的是「你這次改動能不能進 main」。

閘誰執行擋什麼
① check.py 的 drift 檢查scripts/check.py 4b 步驟,filecmp.dircmp 逐檔比對agent 包的 skill 副本跟 vertical 真本內容不一致(忘了 sync,或直接改了副本)
② pre-commit 版本 bump.githooks/pre-commit 呼叫 scripts/version_bump.py --applyplugin 改了卻沒帶版號往前走 — 每次 commit 自動幫你把改動到的 plugin patch-bump 到「比 main 上的版本恰好多 1」,同一分支重複 commit 只會 bump 一次(idempotent),不會每 commit 疊加
③ version 是更新遞送的閘.claude-plugin/plugin.json 的 version 欄位本身Claude Code 只有在偵測到 version 變了,才會把新內容重新推給已安裝的使用者 — 沒 bump 版號,劇本改對了,已安裝的銀行同仁也收不到更新
version-bump 沒有一次到位第一次跑 check.py 時它會自己執行 git config core.hooksPath .githooks 把 hook 接上,所以「忘了裝 hook」這個問題只會發生在還沒跑過一次 check.py 的全新 checkout。.github/workflows/version-bump.yml 這個 GitHub Action 是後備閘 — 沒裝 hook 的 contributor 到了 PR 階段一樣會被攔下來。要跳過單次 commit 的 hook 用 git commit --no-verify,但這只跳過 bump,不代表 CI 的 --check 也會放行。

5. 銀行常見客製清單

把前面四層對照到銀行實際會遇到的客製需求,一次列表方便查:

客製需求屬於哪一層動哪個檔範例
語言(輸出報告改中文)① 系統提示agents/<slug>.md 的產出描述段落「Break list」改成中文欄位名輸出
在地法規(洗錢防制法、個資法)② Skill對應 SKILL.md 的規則段落kyc-rules 加金管會/法務部名單來源(見範例二)
科目表(chart of accounts)② Skillgl-recon 等對帳類 SKILL.md加台灣銀行科目代碼對照表(見範例一)
簽核流程(誰是最終核准人)① 系統提示 + ④ CMA## Workflow 的停點 + callable_agents 分工escalator 改成先給分行經理再給總行法遵
輸出格式(內部報表樣式)② Skillxlsx-author/audit-xls 等格式類 SKILL.md套用本行既有 Excel 範本欄位順序
本章重點
  • 先問「產出的東西是不是同一種、信任邊界有沒有變」再決定調整還是開新 agent — 不是看改動大小。
  • 四個可調層面:系統提示(agents/<slug>.md,兩邊生效)、Skill(vertical-plugins/ 真本 → sync → check)、模型(Layer 1/2/3,多數需求只是 Layer 3 換假設)、CMA 專屬結構(agent.yaml 的 append/tools/steering-examples)。
  • 改 Skill 永遠是真本→sync→check 三步,直接改 agent 包副本會被 check.py 的 drift 檢查擋下。
  • plugin 版有掛、CMA 版沒掛是模型層特有的落差,補 subagent yaml 才會兩邊一致。
  • 三道安全閘(drift / pre-commit 版本 bump / version 遞送閘)管的是「改動能不能進 main、能不能送到已安裝使用者手上」,跟執行期的能力閘/schema 閘/交棒閘是兩回事。

6. 測驗

Q:你想讓 kyc-screener 的雲端(CMA)版也一起套用新的洗錢防制法名單判斷邏輯,應該怎麼做?
Skill 的真本永遠在 vertical-plugins/,agent 包(plugin)底下的副本只是 vendored copy。改真本後跑 sync-agent-skills.py 讓所有 bundle 到這支 skill 的 agent 包同步更新,再跑 check.py 確認沒有 drift、參照完整 — 因為 CMA 的 agent.yaml 也是透過 skills: - { from_plugin: ... } 指向同一個 agent 包,plugin 和 CMA 會同時吃到這次修改,不需要分別改兩邊。直接改副本會被 check.py 的 drift 檢查擋下;system.append 和 steering-examples.json 都不影響 skill 內部的判斷邏輯。
