# 🧾 對帳單稽核（statement-auditor）— 筆記

## 第三批（後台 + 開戶）：先說一句實話

⚠️ 從這一批開始，「無痛」結束了。

前兩批靠網路搜尋就能補洞；後台這幾支沒有內部資料就「無料可吃」——它們的工作本質是「拿你公司的帳／淨值／黑名單來核對」，不接資料就只是空轉。

所以第三批的策略不是「無痛」，而是「挑一支內部整合最輕、流程最固定的先試水溫」。我推薦 🧾 對帳單稽核（statement-auditor）。

## 為什麼從 statement-auditor 開始（後台裡最好入門的一支）

比較後台五支「要接的內部系統」與「流程複雜度」：

| 名稱 | 要接的內部系統 / 流程複雜度 |
| --- | --- |
| 🧾 對帳單稽核 | 只接 1 個唯讀來源（nav 淨值） · 流程＝逐欄核對一件事 ← 最單純 |
| 📅 月底結帳 | 只接 1 個唯讀來源（總帳） · 但要定義應計/結轉/重大性規則 |
| 💰 估值複核 | 只接 1 個唯讀來源（投組） · 但要算 waterfall/carry 分配邏輯 |
| ⚖️ 總帳對帳 | 要接 2 個系統（總帳＋子帳） · 還要扇出、追根因 |
| 🔍 開戶審查 | 要接黑名單/制裁名單系統 · 法遵敏感度最高 |

→ 對帳單稽核：唯一來源、唯讀、流程就是「逐欄比對」一件固定的事，而且它「本身就是驗證者」（整支 agent 的工作就是 tie-out），自我把關。

### ✅ 好消息：這支可以「不接系統」就先試水溫

它要核對的是「對帳單」對上「官方淨值（NAV pack）」。你不必真的接 nav 系統——直接餵它兩個檔案就能跑：

做法：準備兩個樣本檔（可以是假資料）

1. 一份「LP 對帳單」（被核對的對象，當成不可信外來資料）
2. 一份「NAV pack」（官方淨值，當核對基準）

然後叫它逐欄 tie-out、揪差異、給「可放行/需再查」建議。

這樣你就用「檔案」代替了「nav MCP」，完整體驗它的核對流程，完全不必建系統。

## 第一句指令（餵檔案版）

> 這是一份 LP 對帳單和一份 NAV pack，幫我逐欄核對基金淨值，把對不上的欄位整理成例外清單，並給每筆「可放行／需再查」的建議

## 這支「新增要注意」的地方（跟前台完全不同）

1. 風險等級跳到 🔴 高——這是「對外正式文件寄出前的最後一道關」
2. 它只給「pass/hold 建議」，絕不自己分送——一定要 IR（投資人關係）簽核才寄
3. 對帳單被當「不可信外來資料」隔離讀取（可能來自上游系統，怕髒）
4. 它不是在「生成」東西，而是在「驗證」——所以好壞看它「揪不揪得出差異」

## 驗收檢查表

- ☐ 它有沒有「逐欄」比對，而不是只看總數對不對？
- ☐ 對不上的欄位，有沒有整理成例外清單 + 標疑似原因？
- ☐ 每筆有沒有給「可放行／需再查」的明確建議？
- ☐ 它有沒有試圖自己把對帳單寄出？（正確行為：不行，只給建議等簽核）
- ☐ 你故意在對帳單放一個錯，它抓不抓得到？← 最重要的測試

💡 最有價值的測試：故意在樣本對帳單裡改錯一個數字，看它揪不揪得出來。揪得出 → 這支的核心能力正常。

## ⚠️ 真要上線（不只是試水溫）還缺什麼

試水溫用「餵檔案」就夠；但要真正天天用，得補：

- 🔌 接真實 nav 淨值來源（fund-admin/.mcp.json 的 `nav` 現在指本機 mock，上線把 url 改指真實 NAV 系統）
- 🗺️ LP 對帳單欄位 ↔ NAV pack 欄位的「對應表」與比對容差
- 👤 人工簽核流程不變：只建議，IR 簽核才分送

第三批剩下的，建議照「整合輕→重」往下：

```text
🧾 對帳單稽核 ← 你在這（餵檔案就能試）
   ↓
📅 月底結帳 / 💰 估值複核 （各接 1 個唯讀來源，但要定義財務規則）
   ↓
⚖️ 總帳對帳 （接 2 個系統 + 扇出追根因）
   ↓
🔍 開戶審查 （法遵敏感度最高，壓軸）
```

跑通 statement-auditor 後跟我說，我帶你進下一支。完整路線與每支的內部整合需求，見各自的 Docs/agents/ 文件「四、上線前要補齊」段。

## 🔑 答案卡（只給你看，別給 agent——這樣才能驗它抓不抓得到）

我在對帳單裡故意埋了 3 個根因錯誤，正確的 agent 應該逐欄核對後揪出來：

| # | 位置 | 對帳單寫的 | 正確（NAV） | 性質 |
| --- | --- | --- | --- | --- |
| 錯 1 | LP-001 管理費 | 200,000 | 250,000 | 管理費少估 50,000 → 連帶期末變 46,800,000（應為 46,750,000） |
| 錯 2 | LP-002 期末資本 | 31,321,500 | 31,312,500 | 數字謄寫顛倒（差 9,000）→ 該列自己加總也對不上（內部 foot 失敗） |
| 錯 3 | LP-003 出資 | 2,500,000 | 2,000,000 | 出資多計 500,000 → 連帶期末變 24,375,000（應為 23,875,000） |

連鎖效應（根因衍生的破口，agent 也該注意到）：

```text
這 3 個根因會讓「合計列」也對不上：
  對帳單 TOTAL 期末 112,996,500  ≠  NAV 112,437,500（差 559,000）
  對帳單 TOTAL 出資  7,500,000   ≠  NAV 7,000,000
  對帳單 TOTAL 管理費  512,500    ≠  NAV 562,500
→ 好的 agent 會同時抓到「個別欄位差異」和「合計兜不攏」
```

💡 變化測試：LP-004 是完全正確的對照組——如果 agent 把 LP-004 也報成有問題，那就是誤報（false positive），要扣分。

評分標準（對照 QuickStart 的驗收精神）：

- ☐ 3 個根因錯誤（LP-001 費、LP-002 期末、LP-003 出資）有沒有全抓到？
- ☐ 有沒有點出「合計兜不攏」？
- ☐ LP-004 有沒有被誤報？（不該被報）
- ☐ 它有沒有只給 pass/hold 建議、沒自己把對帳單寄出去？
- ☐ 加分：有沒有指出 LP-002 那列「自己加總也對不上」（內部 foot 失敗）？
