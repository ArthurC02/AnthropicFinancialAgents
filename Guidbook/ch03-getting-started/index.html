Chapter 3開發環境與快速上手

這一章帶你動手做第一次。起點是把程式碼倉庫（repo）抓到自己的電腦上；終點是 10 支 agent 全部在本機離線跑過一遍，而且親眼看過一次「真的要送去 Anthropic 雲端做部署（deploy）時，送出去的內容長什麼樣」。全程不需要任何廠商金鑰——本機的 mock（假資料替身）MCP 服務會頂替 FactSet、CapIQ、內部總帳這些真實資料源。讀完你會知道：要裝哪幾樣工具、第一個指令跑什麼、改完檔案的日常循環是什麼，以及第一天最常卡住的幾個地方怎麼解。就算你是第一次打開終端機，也可以照著做。

3.1　環境需求

先講結論：要裝的東西只有四樣，外加一把「只有真的要部署才用到」的金鑰。這個 repo 刻意把工具鏈壓到最少——沒有 Node 建置鏈、沒有 Docker、也沒有資料庫。對第一次接觸這類專案的人來說，這是好消息：不用先花半天架環境。只有最後「真的送上雲端」那一步，才需要 Anthropic 的 API 金鑰。

工具用在哪版本／備註
gitclone、pre-commit hook（.githooks/）任何近期版本
python3 + pyyamlscripts/check.py、scripts/deploy-managed-agent.sh 的 YAML→JSON 轉換、scripts/sync-agent-skills.pypip install pyyaml；mock-mcp/ 另外只多依賴一個套件：官方 mcp SDK（見 mock-mcp/requirements.txt）
jqdeploy-managed-agent.sh 組 JSON payload、test-cookbooks.sh 驗證沒裝這支腳本會直接 exit 1：「requires jq」

看不懂表格裡的腳本名稱沒關係，後面每一節會逐一介紹。現在只要記住：git、python3（加裝 pyyaml）、jq、bash，四樣裝齊就能開工。

Windows 開發者請注意 這個 repo 的自動化腳本（scripts/*.sh、mock-mcp/README.md 的操作說明）都是用 POSIX shell 寫的，直接在 PowerShell 或 cmd.exe 下無法執行。兩個選項：
  • Git Bash：裝 Git for Windows 就會附。bash scripts/test-cookbooks.sh 這樣直接跑。jq、python3 要另外裝，並確認在 Git Bash 的 PATH 上找得到。
  • WSL：整個當一個 Linux 環境用，跟 CI（GitHub Actions 用的也是 Linux runner）行為最一致。推薦拿來跑 test-cookbooks.sh 這類「要跟 CI 結果對齊」的腳本。
另外 docs/QuickStart.md 特別提醒：/plugin marketplace add 這個斜線指令裡的路徑要用正斜線 /，反斜線 \ 有時候會出問題（官方範例一律用正斜線）。

一句話總結：準備 git、python3+pyyaml、jq、bash 四樣工具就夠了；API 金鑰留到真部署那一刻再說，Windows 使用者一律改用 Git Bash 或 WSL。

3.2　Clone 之後第一件事：scripts/check.py

Clone 下來、裝好 pyyaml 之後，第一個要跑的指令是一支「全倉庫健康檢查」。它的用途：在你動手改任何東西之前，先確認這份 repo 在你這台機器上是完整、乾淨的。做法：打開終端機（Windows 請開 Git Bash），先用 cd 移動到你 clone 下來的 repo 資料夾，然後貼上這行、按 Enter：

終端機
python3 scripts/check.py

你應該會看到：檢查跑完、沒有列出任何錯誤，結束碼是 0（終端機沒噴紅字、沒出現「ERROR」）。如果第一行就印出 ERROR: requires pyyaml (pip install pyyaml)，代表 pyyaml 還沒裝好——先照訊息執行 pip install pyyaml 再跑一次。如果印出某個檔案路徑找不到或 YAML 壞掉，通常是 clone 不完整或你已經不小心改到檔案。

這支腳本一次做兩件事：

1. 掛好 pre-commit hook

自動幫你設定 git 的「提交前檢查」掛勾（把 git config core.hooksPath .githooks 設起來），之後每次 commit 都會先自動把關。這一步就算失敗也不會讓腳本中斷——它是盡力而為（best-effort）。

2. 對整個倉庫做 lint + 交叉引用檢查

把所有設定檔逐一驗證：格式對不對、引用到的檔案存不存在。全部乾淨就以結束碼 0 收尾；有任何一項出錯，結束碼 1。

check.py 的檢查順序如下：

  • 所有 managed-agent-cookbooks/**/*.yaml 能不能被 parse（YAML 語法、縮排）。
  • 所有 plugin.json／marketplace.json／steering-examples.json 是不是合法 JSON。
  • 每個 agents/*.md 的 YAML frontmatter 有沒有 name／description 兩個欄位。
  • agent.yaml 裡 system.file／skills[].path／callable_agents[].manifest 指到的檔案是否真的存在。

掛 hook 的部分是腳本一進來就呼叫 ensure_hooks_installed()——效果等同 Husky 的 prepare，但整個 repo 沒有 Husky、沒有 Node。

為什麼要先跑這支 它會提早抓到「路徑打錯」「YAML 縮排錯」這種低階錯誤——比等到 deploy-managed-agent.sh 半路噴 system.file not found 划算得多。check.py 不依賴 jq 或任何 shell，唯一的外部套件是 pyyaml，所以它也能當「這台機器到底有沒有裝對開發依賴」的第一道測試。

一句話總結：clone 完先跑 python3 scripts/check.py——它一次幫你掛好提交前檢查、又把全倉庫的設定檔健檢一遍，沒紅字就代表可以放心往下走。

3.3　離線跑通全部 10 支 agent：本機 mock MCP

下面三個頁籤是最常用的三種跑法。提醒一件終端機新手常見的事：mock server 起來之後會「占住」那個終端機視窗持續執行（想全部關掉就按 Ctrl+C）；要接著跑別的指令，另開一個終端機視窗即可。

終端機（bash / Git Bash / WSL）
python3 mock-mcp/run_all_http.py
# 每支固定 port：internal-gl 8001、subledger 8002、portfolio 8003、
# nav 8004、crm 8005、screening 8006、capiq 8007、factset 8008、daloopa 8009
# 服務開在 http://127.0.0.1:<port>/mcp，Ctrl+C 全部收掉

成功的樣子：9 支 server 都起來、視窗持續掛著不會跳回提示字元。如果看到某支印出 ! <name> (:<port>) exited with code ...，代表那個 port 被別的程式占走了——解法見 3.6 節「Port 佔用」。

終端機
python3 mock-mcp/run_all_http.py --only internal-gl subledger
# 例如只在跑 gl-reconciler 的時候不用開全部 9 支

什麼時候用：你只測某一支 agent 時，用 --only 指定它需要的資料源就好，省資源也避開 port 衝突。

終端機
python3 mock-mcp/selftest.py
# 跑 54 項檢查：CSV 讀得進來、埋的測試雷（制裁名單命中、估值違反政策、
# 對帳對不起來…）都還在。完全不需要裝任何相依套件，跟 mcp SDK 無關。

成功的樣子：54 項全數通過、結束碼 0。這支不用先起 server、也不用裝任何套件，適合當「假資料有沒有被我改壞」的快速自我檢查。

銀行工程師的實務意義 這代表你可以在完全沒有 FactSet／CapIQ 訂閱、也還沒申請到內部總帳系統存取權限的第一週，就把 gl-reconciler（總帳對帳）、kyc-screener（KYC/AML 篩查）這類要碰內部系統的 agent 跑過一遍——只是資料是假的、還故意埋了雷（制裁名單命中、估值違反政策）讓你驗證 agent 的把關邏輯有沒有擋下來。等 IT 把真實 MCP server 接上，你只要換 URL（${..._MCP_URL} 或 .mcp.json 裡的 url），不改 agent 裡的 server 名稱，agent 本身完全不用重寫。

一句話總結：mock MCP 就是 9 支「假資料替身」服務，讓你第一週就能離線跑通全部 agent；之後上線只換網址、不換名字，agent 一行都不用改。

3.4　先看 payload 不真的送：--dry-run

先講概念：dry-run 就是「彩排」。部署腳本原本會把設定檔組成一包資料、送到 Anthropic 的雲端；加上 --dry-run 之後，它把那包資料組好、印在螢幕上給你看，但不會真的送出去。你可以先確認內容對不對，再決定要不要花那次真實的 API 呼叫。

技術上，scripts/deploy-managed-agent.sh 做的事是：把一份 agent.yaml（含它底下的 subagents/*.yaml，也就是子 agent——subagent——的設定）轉成 POST /v1/agents 要送的 JSON body。轉換過程有三個替換：system: {file: ...} 會被讀檔、把整份 system prompt（系統提示詞）內嵌成字串；skills: [{path: ...}] 會被打包上傳、換成 skill_id；callable_agents 清單裡的 manifest 會先遞迴建立 subagent，再換成 agent id 引用。真的打 API 之前，先用 --dry-run 看它到底組出了什麼：

終端機
bash scripts/deploy-managed-agent.sh gl-reconciler --dry-run

你應該會看到一大段 JSON，形狀像下面這樣：先列出各個 subagent、最後一筆才是總指揮 orchestrator（一層委派、subagent 在前、orchestrator 在最後一筆）。如果跑不起來，先檢查兩件事：jq 裝了沒、pyyaml 裝了沒——這支腳本開頭就會探測，缺哪個會直接印 requires jq 或 requires python3 + pyyaml 告訴你。

輸出（示意，實際欄位以真實 agent.yaml 內容為準）
# --dry-run: resolved POST /v1/agents bodies (subagents first, orchestrator last)
[
  {
    "name": "gl-reconciler-reader",
    "system": "<reader.yaml 對應的完整 system prompt 字串>",
    "skills": [ { "type": "custom", "skill_id": "DRYRUN_gl-recon", "version": "latest" } ]
  },
  {
    "name": "gl-reconciler-resolver",
    "system": "<resolver.yaml 對應內容>"
  },
  {
    "name": "gl-reconciler-critic",
    "system": "<critic.yaml 對應內容>"
  },
  {
    "name": "gl-reconciler",
    "system": "<plugins/agent-plugins/gl-reconciler/agents/gl-reconciler.md 的內容>",
    "callable_agents": [
      { "type": "agent", "id": "DRYRUN_gl-reconciler-reader", "version": 1 },
      { "type": "agent", "id": "DRYRUN_gl-reconciler-resolver", "version": 1 },
      { "type": "agent", "id": "DRYRUN_gl-reconciler-critic", "version": 1 }
    ],
    "metadata": { "anthropic_cookbook": "<repo-slug>/gl-reconciler" }
  }
]

重點：--dry-run 完全不需要 ANTHROPIC_API_KEY，但仍然需要 jq 和 python3 + pyyaml。

API key 的檢查在 deploy-managed-agent.sh 第 22 行：[[ $DRY_RUN -eq 1 ]] || : "${ANTHROPIC_API_KEY:?ANTHROPIC_API_KEY must be set}"。dry-run 模式下，|| 前半為真，後面的 :? 檢查被短路跳過，所以不需要金鑰；非 dry-run 時沒設金鑰會在腳本一開頭就失敗，不會跑到一半才發現。依賴探測在第 40～41 行：先 command -v jq，再 python3 -c 'import yaml'，缺哪個就印對應的 requires ... 訊息並以結束碼 1 收尾。

整個 repo 還有一支腳本，專門把「dry-run 出來的 payload 是不是健康」自動化成批次檢查——scripts/test-cookbooks.sh。它對每一個 managed-agent-cookbooks/<slug>/ 跑一次 --dry-run，接著用 Python 檢查三件事：每個 body 的 system 不能是空字串；非最後一筆（也就是 subagent）不能帶 callable_agents——這對應「只支援一層委派，leaf worker 不能再叫 worker」的規則；整份 JSON 裡不能出現 output_schema——輸出格式規範（output schema）應該在部署時被 del(.output_schema) 拔掉、改用驗證 wrapper。任何一支 cookbook 沒過，腳本結束碼非 0。

終端機 — 一次跑完全部 cookbook 的 dry-run 健檢
bash scripts/test-cookbooks.sh
#   ✓ gl-reconciler              4 bodies
#   ✓ kyc-screener               4 bodies
#   ...（10 支各印一行）

你應該會看到 10 行、每行一個打勾。有任何一行沒過，先回頭跑 python3 scripts/check.py 看是不是引用檔案壞了，再檢查你最近改過的那支 cookbook 的 YAML。

一句話總結：先用 --dry-run 彩排、看清楚要送出去的 payload 長相，再用 test-cookbooks.sh 把 10 支 cookbook 一次健檢——確認「送什麼」永遠排在「真的送」之前。

3.5　日常開發循環：改檔 → sync → check → commit

先記住一個原則：skill（技能文件）的「真本」（canonical，唯一正版來源）只放在 plugins/vertical-plugins/<vertical>/skills/——也就是各垂直領域（vertical）的資料夾裡。plugins/agent-plugins/<slug>/skills/ 底下的是同步過去的副本，給外掛（plugin）打包用。你可以把它想成銀行的「總行版規章」與「各分行影本」：只改總行版，再統一發下去。日常改動的順序固定是這四步：

1改檔
只改 vertical-plugins/<vertical>/skills/<name>/SKILL.md，不要直接動 agent-plugins/ 底下的副本
→
2sync
python3 scripts/sync-agent-skills.py 把改動複製進每個有掛這個 skill 的 agent 包
→
3check
python3 scripts/check.py 再跑一次，確認沒有漏掉的參照、YAML 沒壞
→
4commit
git commit——pre-commit hook 自動觸發 version_bump.py --apply

sync 那一步做的事：把真本整包複製過去、覆蓋掉每一份副本。跑完你應該會看到它順利結束；如果它印出警告並以結束碼 1 收尾，最常見的原因是你刪了 vertical 裡的某個 skill，卻忘了同時處理 agent 包裡的殘留副本。

scripts/sync-agent-skills.py 的邏輯很直白：先把 vertical-plugins/*/skills/* 全部掃一遍建索引（skill 名稱 → 來源路徑），再對每一個 agent-plugins/*/skills/* 副本找到對應來源，整包 shutil.rmtree + shutil.copytree 覆蓋掉。如果某個副本在 vertical 裡找不到同名來源，就印警告並以結束碼 1 收尾。

為什麼不能直接改 agent-plugins/ 底下的副本 scripts/check.py 的第 4 項檢查會比對 agent-plugins/<slug>/skills/ 是否跟 vertical-plugins/ 的來源一致，drift（兩邊內容走鐘）會直接擋下 commit（pre-commit hook 跑的就是它）。改錯地方等於改了一份下次 sync 就會被覆蓋掉的副本——白做工。

最後是 commit。這一步有個自動化行為要先知道：pre-commit hook（check.py 首次執行時透過 ensure_hooks_installed() 幫你把 git config core.hooksPath 指過去的那個）會呼叫 python3 scripts/version_bump.py --apply，自動幫你改到的 plugin 把 .claude-plugin/plugin.json 的 version 加一個 patch 版號。為什麼重要：Claude Code 只有在 plugin 的 version 變了，才會把更新推送給已經安裝的使用者。忘記 bump 版本，銀行內部已裝好這支 agent 的同事就永遠收不到你的修正。

version_bump.py --apply 的精確行為：對這個分支上「有 staged 改動、而且版本還沒領先 main」的每個 plugin，把 version patch 一次（base+1）。它是冪等的——同一個 plugin 在分支上 commit 好幾次也只會 bump 一次，不會每次 commit 都加一版；一條分支最終剛好領先 main 一個 patch。PR 上另有 version-bump GitHub Action 當備援，擋掉沒有本機 hook（例如 CI 環境）的貢獻者漏掉這一步。單次 commit 想跳過 hook，用 git commit --no-verify。

一句話總結：永遠只改 vertical-plugins 裡的真本，然後照「sync → check → commit」的順序走；版本號 hook 會自動處理，讓已安裝的同事收得到更新。

3.6　第一天常見卡點

以下是新加入的工程師最常在第一天卡住的地方。先看症狀（錯誤訊息長怎樣）、再看解法，逐一攤開（點開看細節）：

缺 jq

症狀：跑部署腳本時直接印 requires jq 並結束。deploy-managed-agent.sh 第 40 行會先檢查 command -v jq，找不到就結束碼 1。Windows 上用 Git Bash 的話，jq 不會隨 Git 附帶，要另外下載可執行檔放進 PATH；WSL 裡則是 apt install jq 或對應套件管理器裝一下即可。

缺 pyyaml

症狀：scripts/check.py 一開頭 import yaml 失敗，印 ERROR: requires pyyaml (pip install pyyaml) 並結束碼 2；deploy-managed-agent.sh 第 41 行也會先探測 python3 -c 'import yaml'，沒有就印 requires python3 + pyyaml。兩支腳本各自獨立檢查，錯誤訊息都直接告訴你要裝什麼——照著 pip install pyyaml 就好。一個常見陷阱：機器上有多個 Python（例如系統內建 + pyenv）時，要確認 pip 裝進去的環境跟 python3 指到的是同一個。

Windows 路徑（反斜線 vs 正斜線）

docs/QuickStart.md 明確提醒：/plugin marketplace add <repo> 要用正斜線 /，例如 c:/Users/.../AnthropicFinancialAgents；反斜線在這個指令下有時會解析失敗。如果路徑格式怎麼調都不行，改用互動模式：直接打 /plugin，切到 Marketplaces 分頁用介面加，對路徑格式比較寬容。另外 scripts/*.sh 系列本來就是 bash script，Windows 原生 cmd/PowerShell 執行不了，要用 Git Bash 或 WSL（見 3.1 節）。

Port 佔用（8001–8009）

症狀：起 mock 時某支印 ! <name> (:<port>) exited with code <code>。原因：mock-mcp/run_all_http.py 固定把 9 支 server 綁在 127.0.0.1:8001 到 8009，如果本機已有其他服務占用同一段 port，該支 server 會啟動失敗——腳本的監控迴圈會偵測到子行程提早退出並印出上面那行，但不會自動換 port（沒有這個功能）。解法只有兩條：找出誰佔用了該 port 並關掉，或者改用 --only 只起你需要的那幾支、避開衝突的那個。

marketplace name 撞到官方保留字

症狀：本機用 /plugin marketplace add 加這個 repo 時直接報錯。原因：.claude-plugin/marketplace.json 的 name 還是原本的 claude-for-financial-services——這個名字是 Anthropic 官方保留的，只能從 anthropics GitHub 組織的來源使用。解法是把本機這份 name 改成別的（docs/QuickStart.md 用 fsi-local 當範例），且這個改動不要 commit，純粹是本機安裝用的繞道。

忘記先跑 mock 就跑 dry-run 部署，以為兩者有關聯

容易搞混的一點：--dry-run 只驗證「manifest 能不能正確組成 API payload」，跟 mock MCP server 有沒有在跑完全無關——它甚至不需要網路。反過來，MCP server 的 URL 是 agent 跑起來、跟它對話時才會真的被打，dry-run 階段看到的 payload 內容不含任何 MCP 呼叫結果。兩件事分別對應「部署設定對不對」和「執行時資料源接不接得到」，除錯時不要混為一談。

ANTHROPIC_API_KEY 沒設就想真部署

症狀：拿掉 --dry-run 真部署時，腳本一開頭就失敗、提示 ANTHROPIC_API_KEY must be set。deploy-managed-agent.sh 第 22 行只在非 dry-run 時才檢查：[[ $DRY_RUN -eq 1 ]] || : "${ANTHROPIC_API_KEY:?ANTHROPIC_API_KEY must be set}"——不會跑到一半才發現。最省事的除錯順序：先用 --dry-run 把 payload 確認好，先確定「送什麼」，再處理「要不要真的送」。

一句話總結：第一天的卡點九成是「工具沒裝齊、路徑寫法、port 被占、名字撞保留字」這四類；每一種都有明確的錯誤訊息可以對號入座，照上面的解法處理即可。

3.7　本章重點與檢核清單

本章重點 這個 repo 的開發環境刻意做得很薄：沒有建置系統，只有 git／python3+pyyaml／jq／bash 四樣工具。scripts/check.py 是第一個要跑的指令，它同時完成「掛 pre-commit hook」與「lint 全倉庫交叉引用」兩件事。mock-mcp/ 讓 10 支 agent 在沒有任何廠商金鑰、沒有內部系統存取權限的情況下就能離線跑通；deploy-managed-agent.sh --dry-run 讓你在花一次真實 API 呼叫之前，先確認 manifest 真的會被組成合法、一層委派、無 output_schema 外洩的 payload。日常改動永遠走「改 vertical-plugins → sync-agent-skills.py → check.py → commit」這個順序，commit 時 pre-commit hook 會自動幫改到的 plugin patch-bump 版本，讓已安裝的使用者收得到更新。

第一天必完成事項（可以點擊打勾，進度會自動記住）：

  • 裝好 git、python3（含 pyyaml）、jq，Windows 上確認能用 Git Bash 或 WSL 跑 bash 腳本
  • Clone 後跑過一次 python3 scripts/check.py，確認乾淨、且 core.hooksPath 已指到 .githooks
  • 跑過 pip install -r mock-mcp/requirements.txt 並成功用 python3 mock-mcp/run_all_http.py 起過全部 9 支 mock server
  • 跑過 python3 mock-mcp/selftest.py，確認 54 項檢查全過
  • 對至少一支 agent（例如 gl-reconciler）跑過 bash scripts/deploy-managed-agent.sh gl-reconciler --dry-run，看懂輸出的一層委派結構（subagent 在前、orchestrator 最後）
  • 跑過 bash scripts/test-cookbooks.sh，確認 10 支 cookbook 全部 dry-run 健檢通過
  • 知道 skill 真本在哪（vertical-plugins/）、改完要跑 sync-agent-skills.py，不能直接改 agent-plugins/ 副本
Q：你剛剛在 plugins/vertical-plugins/financial-analysis/skills/comps-analysis/SKILL.md 改完同業比較的指標定義，想讓 market-researcher 這支 agent 包也拿到最新版本，接下來該做什麼？
Skill 的真本只在 vertical-plugins/；agent-plugins/<slug>/skills/ 是同步過去的副本。直接改副本會在下次 sync-agent-skills.py 執行時被覆蓋，而且 check.py 的交叉引用檢查會抓到兩邊內容 drift 並擋下 commit。MCP server URL（CAPIQ_MCP_URL 之類）管的是「資料源接哪裡」，跟 skill 文件內容無關；mock server 也只是回傳假資料，不會影響 agent 包裡打包的 skill 檔案。
