# 🗂️ Agent 總覽 — 十個助理，一條完整的工作鏈

> 這十個 AI 助理不是各跑各的工具，而是把一家金融公司「**贏得客戶 → 開戶 → 基金日常營運 → 對出資人交代**」整條 pipeline 串起來的一套班底。下面先用一張表讓你看懂每個助理，再用一個實際故事串起來，最後聊導入評估。
>
> 風險標記：🟢 低、🟡 中、🔴 高（越靠近「錢」跟「法規」，風險越高、把關就越嚴）。

---

## 一、十個助理一覽（在做什麼・什麼時候用・風險）

### 🟦 客戶經營與顧問（前台）
| 助理 | 用途 | 什麼時候用 | 風險 |
|---|---|---|---|
| 🤝 **會前準備**（meeting-prep-agent） | 開會前先把客戶關係、持倉跟相關市場動態彙整起來，產出給顧問的會前資料包 | 見客戶前 | 🟢 |
| 🎯 **投行提案**（pitch-agent） | 蒐集同業比較跟估值，產出含估值模型和品牌簡報的提案包 | 要對一個對象做第一版提案 | 🟡 |

### 🟩 研究與建模（前台）
| 助理 | 用途 | 什麼時候用 | 風險 |
|---|---|---|---|
| 🔭 **市場研究**（market-researcher） | 研究產業概況、競爭對手、同業比較，產出研究報告 | 要搞懂一個產業或主題 | 🟡 |
| 📊 **財報追蹤**（earnings-reviewer） | 公司發完財報後解讀法說會、更新模型，產出財報後的觀點筆記 | 追蹤中的股票發財報了 | 🟡 |
| 🧮 **估值建模**（model-builder） | 從零建估值模型（現金流折現、槓桿收購、三大報表等），產出可以拿來做決策的試算表 | 要一份全新的估值模型 | 🟡 |

### 🟧 基金行政與財務（後台）
| 助理 | 用途 | 什麼時候用 | 風險 |
|---|---|---|---|
| ⚖️ **總帳對帳**（gl-reconciler） | 比對總帳跟明細帳找出差異、追原因，產出例外報告 | 每天／月底要對帳 | 🔴 |
| 📅 **月底結帳**（month-end-closer） | 估列未付費用、結轉餘額、說明差異，產出月結包 | 每月底結帳 | 🔴 |
| 💰 **估值複核**（valuation-reviewer） | 季末複核投資標的估值、算收益分配，產出給出資人的報告 | 季末估值 | 🔴 |
| 🧾 **對帳單稽核**（statement-auditor） | 對帳單寄給出資人之前，逐欄核對基金淨值、揪差異 | 對帳單寄出前的最後把關 | 🔴 |

### 🟥 開戶與審查（中台）
| 助理 | 用途 | 什麼時候用 | 風險 |
|---|---|---|---|
| 🔍 **開戶審查**（kyc-screener） | 解析開戶文件、套反洗錢規則、比對黑名單，產出給合規人員的審查報告 | 新客戶開戶或定期複審 | 🔴 |

> 共通點：**每個助理都只負責「起草、建議」，不會自己下決定**——最後一定有人 review、簽核。

## 二、一個實際故事：晨曦資本的一季

> 晨曦資本是一家綜合金融公司，既做投資銀行顧問、也經營自己的基金。這一季，它要替新基金找一個大型出資人（把錢交給基金去投資的機構），同時照常營運基金。我們來看十個助理各自在哪一刻登場。

```
時間 ───────────────────────────────────────────────────────────►

 ① 前台·贏得業務       ② 中台·開戶     ③ 後台·基金日常     ④ 後台·季末交代
 ────────────────      ──────────      ──────────────      ──────────────
 🤝 會前準備           🔍 開戶審查      ⚖️ 總帳對帳         💰 估值複核
 🔭 市場研究                           📅 月底結帳         🧾 對帳單稽核
 📊 財報追蹤
 🧮 估值建模
 🎯 投行提案
```

### ① 前台：把潛在出資人變成有興趣的客戶
1. **🤝 會前準備**　顧問下週要見一個大學捐贈基金（潛在出資人），助理會前先備好「往來歷史＋持倉＋相關市場動態＋建議議程」。
2. **🔭 市場研究**　會中對方問：「AI 基礎建設這主題值得投嗎？」→ 助理隔天就交出產業概覽＋競爭格局＋同業比較＋值得關注的名單。
3. **📊 財報追蹤**　名單裡有一檔追蹤中的股票剛好發財報 → 助理讀法說會、更新模型、草擬觀點筆記，讓看法保持最新。
4. **🧮 估值建模**　對方想深入評估其中一家公司 → 助理從零把現金流折現跟槓桿收購估值建好，供投資判斷。
5. **🎯 投行提案**　同一時間，投行端要對一個併購對象提案 → 助理自動蒐集同業比較、建估值、產出品牌提案簡報。

### ② 中台：資金要進場，先開戶
6. **🔍 開戶審查**　出資人決定出資 → 助理解析開戶文件、跑開戶跟反洗錢審查、比對制裁名單跟政治人物名單，整理成審查包丟給合規人員簽核。
   > ⛔ 這一關沒過，後面都不會發生——開戶就是資金進場的閘門。

### ③ 後台：基金開始營運
7. **⚖️ 總帳對帳**　資金進場後，每天比對總帳跟明細帳，揪出差異、追原因。
8. **📅 月底結帳**　每到月底，估列未付費用、結轉餘額、說明差異，組裝結帳包。

### ④ 後台：季末對出資人交代
9. **💰 估值複核**　季底複核投資標的估值、算收益分配——管理團隊的績效分成跟出資人各分多少，順便備好出資人報告。
10. **🧾 對帳單稽核**　對帳單寄出前的最後一道把關——逐欄核對基金淨值、揪差異、給「可放行／需再查」。
    > ✅ 過了這關，報告才寄給出資人——一季的循環就收尾了。

## 三、五條貫穿全部的設計模式（從 10 支 source 歸納）

**模式 ①　Write 權跟著風險走——風險越高，越不讓 agent 自己動手**
```
🟢🟡 前台／研究  → 產出只是內部草稿         → 助理可以自己存檔（較寬鬆）
🔴   後台／合規  → 牽涉法規與帳目正確性     → 助理不能自己存檔，動作嚴格切開、分權
```

**模式 ②　髒 reader vs 乾淨 puller（決定要不要防注入）**

- 碰外來髒文件的（逐字稿／護照／對手對帳單／發票／LP 單／GP 包／客戶信／第三方報告）→ 那個 reader 一律 `mcp_servers:[]` ＋ output_schema 鎖死，注入指令活不下來。
- input 全走可信 MCP 的（如 model-builder 的 data-puller）→ schema 只是「乾淨結構化交棒」，不是防注入。

**模式 ③　schema 把「分類」鎖成 enum**

讓 reader 連選項都不能自由發揮：gl 的 `suspected_cause`、valuation 的 `method`、gl reader 的 `status` 都是固定 enum——注入想塞自由文字也塞不進來。

**模式 ④　驗證方式隨任務性質變**

| Agent | 驗證設計 |
|---|---|
| model-builder | 獨立 auditor 重查那份檔 |
| gl-reconciler | critic 拿可信源重算，濾誤報（對抗式） |
| month-end-closer | 自我 foot check（期初＋變動＝期末），無 critic |
| statement-auditor | agent 本身就是驗證者（tie-out 即工作） |
| valuation-reviewer | 對照政策（marks vs policy） |
| earnings／pitch | 自查（audit-xls／ib-check-deck QC） |
| kyc-screener | 規則引擎＋篩查，最後合規官決定 |

> 📎 這些模式的 source 細節與「怎麼改才不會破壞它們」見 [TechSummary.md](TechSummary.md) 跟 [Customizing.md](Customizing.md)。

**模式 ⑤　人永遠最後拍板，簽核層級隨風險升高（顧問 → 分析師 → 控制員 → IR → IR+CCO）**
```
所有助理都只「起草、建議」，從不自己決定：
 開戶 → 合規人員決定   過帳 → 控制員核准   發布 → 分析師簽核   寄送 → 主管簽核
```

## 四、導入評估（風險・成本・先後）

> 各助理的完整分析見各別文件「五、導入評估」；這裡只給整體對照跟導入順序。

| 助理 | 導入風險 | 導入成本 | 適用單位 | 主要使用者 | 最適合的場景 |
|---|---|---|---|---|---|
| 🤝 會前準備 | 🟢 低 | 🟡 中 | 財富管理前台 | 財富顧問 | 例行客戶會議前的資料準備 |
| 🔭 市場研究 | 🟢 低 | 🟡 中 | 研究部／投資團隊 | 研究分析師 | 不熟的產業/主題要快速上手 |
| 📊 財報追蹤 | 🟡 中 | 🟡 中 | 股票研究部 | 股票研究分析師 | 財報季要同時追很多檔 |
| 🧮 估值建模 | 🟡 中 | 🟡 中 | 投行／研究部 | 建模專員／分析師 | 從零建一份乾淨的估值模型 |
| 🎯 投行提案 | 🟡 中 | 🟡 中偏高 | 投資銀行前台 | 資深投行人員 | 初次接觸客戶的第一版提案 |
| 🔍 開戶審查 | 🔴 高 | 🔴 高 | 法令遵循部 | 合規人員 | 新客戶開戶的初步審查 |
| ⚖️ 總帳對帳 | 🔴 高 | 🔴 高 | 基金行政後台 | 對帳控制員 | 每天例行的帳目核對 |
| 📅 月底結帳 | 🔴 高 | 🔴 高 | 基金行政後台 | 控制員 | 每月例行的月結打底 |
| 🧾 對帳單稽核 | 🔴 高 | 🔴 高 | 基金行政部 | 基金會計 | 對帳單分送前的把關 |
| 💰 估值複核 | 🔴 高 | 🔴 高 | 基金行政／會計 | 基金會計主管 | 季末估值複核與收益分配 |

**部門與角色:誰下指令 → 誰簽核**

> 上表只給「主要使用者」一個角色;實務上每支都是「**建立者下指令 → 另一個角色簽核**」。這張表把完整的角色鏈攤開(原本要開 10 個別檔才看得到)。

| 助理 | 部門 | 誰下指令 → 誰簽核 |
|---|---|---|
| 🤝 會前準備 | 財富管理／私人銀行前台 | 財富顧問(下指令＋會前 review)→ 顧問自己對客戶 |
| 🎯 投行提案 | 投行前台(併購／企業融資)、Coverage 產業組 | 分析師／經理建 → **MD／資深 banker 兩次簽核** |
| 🔭 市場研究 | 賣方／買方研究部、投資團隊 | 研究分析師(下指令＋review)→ 分析師核准 |
| 📊 財報追蹤 | 股票研究部 | 股票分析師建 → **資深分析師簽核才發** |
| 🧮 估值建模 | 投行、研究、PE／併購、corp dev／FP&A | 建模專員／分析師建 → banker／PM 核准 |
| ⚖️ 總帳對帳 | 基金行政後台、帳務控制 | 對帳控制員建 → **控制員簽核**(帳務主管核容差) |
| 📅 月底結帳 | 基金行政、總帳／結帳部 | 結帳會計建 → **控制員核准才過帳**(唯一可 post) |
| 🧾 對帳單稽核 | 基金行政、基金會計、IR | 基金會計建 → **IR 簽核才分送** |
| 💰 估值複核 | 基金行政／基金會計、PE 後台 | 基金會計主管建 → **IR ＋ CCO 雙簽才分送** |
| 🔍 開戶審查 | 法遵部、AML 專責、開戶作業 | 開戶人員送件 → **合規官簽核才核准** |

> 規律:**前台 = 同一個人下指令兼 review;中後台 = 建立者 ≠ 簽核者(分權)**,而且越靠近「錢／法規」,簽核者層級越高(控制員 → IR → CCO)。

**一條很清楚的規律：前台「低風險、低門檻」，後台「高風險、高門檻」**

```
            導入成本低 ──────────────────► 導入成本高
 風險低  │                   🤝 會前準備    🔭 市場研究
 風險中  │                   📊 財報追蹤  🧮 估值建模  🎯 投行提案
 風險高  │                   🔍 開戶審查  ⚖️ 總帳對帳  📅 月底結帳
         │                   🧾 對帳單稽核  💰 估值複核
```
> 為什麼後台都落在右上角（又貴又難）：因為它們要接的是公司**內部系統**（總帳、明細帳、基金淨值、投資組合、黑名單、客戶系統）。repo 內建的本機 mock 讓你能先**離線 demo**，但要正式上線，就得把那些 mock 改指公司真實系統（成本高）；又牽涉法規跟帳目正確性，錯不得（風險高）。前台靠的是外部市場資料（向資料商訂閱就有），產出又只是內部草稿（有人 review 能攔錯）。

**導入順序建議（先挑簡單見效的，再啃硬骨頭）**

```
第一批　先導入：研究與顧問四支（🤝 會前準備、🔭 市場研究、📊 財報追蹤、🧮 估值建模）
        風險低到中、成本中等，多靠外部訂閱資料、不必接內部系統，最快見效、試錯成本低。

第二批　接著導入：🎯 投行提案
        風險中、但成果會對外，且要大量客製投行的簡報範本與檢查規則，工程量比第一批大。

第三批　最後導入：中後台五支（後台四支：⚖️ 總帳對帳、📅 月底結帳、🧾 對帳單稽核、💰 估值複核＋中台一支：🔍 開戶審查）
        省下的人工最多、價值最高，但門檻也最高：要接內部系統、對齊法遵與財務流程、
        建立可追查的操作紀錄。建議在前兩批累積信任後再上。
```
