Chapter 3開發環境與快速上手

這一章把你從「clone 下來」帶到「10 支 agent 全部離線跑過一次、也看過一次真正要送去 Anthropic 雲端的部署 payload 長什麼樣」。全程不需要任何廠商金鑰——本機 mock MCP 會頂替 FactSet / CapIQ / 內部總帳這些資料源。讀完你會知道：裝什麼工具、第一個指令要跑什麼、改完程式碼的日常循環是什麼，以及第一天最常卡住的幾個地方怎麼解。

3.1　環境需求

這個 repo 沒有 Node 建置鏈、沒有 Docker、也沒有資料庫——工具鏈刻意壓到最少。你需要裝的東西只有這幾樣：

工具用在哪版本／備註
gitclone、pre-commit hook（.githooks/）任何近期版本
python3 + pyyamlscripts/check.py、scripts/deploy-managed-agent.sh 的 YAML→JSON 轉換、scripts/sync-agent-skills.pypip install pyyaml；mock-mcp/ 另外只多依賴一個套件：官方 mcp SDK（見 mock-mcp/requirements.txt）
jqdeploy-managed-agent.sh 組 JSON payload、test-cookbooks.sh 驗證沒裝這支腳本會直接 exit 1：「requires jq」
bash所有 scripts/*.shWindows 請用 Git Bash 或 WSL，見下方提醒
ANTHROPIC_API_KEY只有「真的部署」（非 --dry-run）才需要deploy-managed-agent.sh 第 22 行：[[ $DRY_RUN -eq 1 ]] || : "${ANTHROPIC_API_KEY:?ANTHROPIC_API_KEY must be set}"——沒設就直接失敗，不會偷偷用假值
Windows 開發者請注意 這個 repo 的自動化腳本（scripts/*.sh、mock-mcp/README.md 的操作說明）都是用 POSIX shell 寫的，直接在 PowerShell 或 cmd.exe 下無法執行。兩個選項：
  • Git Bash：裝 Git for Windows 就會附。bash scripts/test-cookbooks.sh 這樣直接跑。jq、python3 要另外裝並確保在 Git Bash 的 PATH 上能找到。
  • WSL：整個當一個 Linux 環境用，跟 CI（GitHub Actions 用的也是 Linux runner）行為最一致，推薦拿來跑 test-cookbooks.sh 這類「要跟 CI 結果對齊」的腳本。
另外 docs/QuickStart.md 特別提醒：/plugin marketplace add 指令裡的路徑要用正斜線 /，反斜線 \ 有時候會出問題（官方範例一律用正斜線）。

3.2　Clone 之後第一件事：scripts/check.py

Clone 下來、裝好 pyyaml 之後，第一個要跑的指令是：

終端機
python3 scripts/check.py

它做兩件事：

1. 掛好 pre-commit hook

一進腳本就呼叫 ensure_hooks_installed()，把 git config core.hooksPath .githooks 設起來（等於 Husky 的 prepare，但沒有 Husky、沒有 Node）。失敗也不會讓腳本中斷——它是 best-effort。

2. 對整個倉庫做 lint + 交叉引用檢查

依序驗證：所有 managed-agent-cookbooks/**/*.yaml 能不能 parse、所有 plugin.json／marketplace.json／steering-examples.json 是不是合法 JSON、每個 agents/*.md 的 YAML frontmatter 有沒有 name／description、以及 agent.yaml 裡 system.file／skills[].path／callable_agents[].manifest 指到的檔案是否真的存在。乾淨則結束碼 0，有錯就結束碼 1。

為什麼要先跑這支 它會提早抓到「路徑打錯」「YAML 縮排錯」這種低階錯誤——比等到 deploy-managed-agent.sh 半路噴 system.file not found 划算得多。check.py 沒有相依 jq 或任何 shell，唯一的外部套件是 pyyaml，所以它可以當「這台機器到底有沒有裝對開發依賴」的第一道測試。

3.4　先看 payload 不真的送：--dry-run

scripts/deploy-managed-agent.sh 把一份 agent.yaml（含它底下的 subagents/*.yaml）轉成 POST /v1/agents 要送的 JSON body：system: {file: ...} 會被讀檔內嵌成字串、skills: [{path: ...}] 會被打包上傳換成 skill_id、callable_agents: [{manifest: ...}] 會先遞迴建立 subagent 再換成 agent id 引用。真的打 API 之前，先用 --dry-run 看它到底組出了什麼：

終端機
bash scripts/deploy-managed-agent.sh gl-reconciler --dry-run

預期輸出形狀（--dry-run 分支：先遞迴建立完子 agent 再印出來，depth-1，subagent 在前、orchestrator 在最後一筆）：

輸出（示意，實際欄位以真實 agent.yaml 內容為準）
# --dry-run: resolved POST /v1/agents bodies (subagents first, orchestrator last)
[
  {
    "name": "gl-reconciler-reader",
    "system": "<reader.yaml 對應的完整 system prompt 字串>",
    "skills": [ { "type": "custom", "skill_id": "DRYRUN_gl-recon", "version": "latest" } ]
  },
  {
    "name": "gl-reconciler-resolver",
    "system": "<resolver.yaml 對應內容>"
  },
  {
    "name": "gl-reconciler-critic",
    "system": "<critic.yaml 對應內容>"
  },
  {
    "name": "gl-reconciler",
    "system": "<plugins/agent-plugins/gl-reconciler/agents/gl-reconciler.md 的內容>",
    "callable_agents": [
      { "type": "agent", "id": "DRYRUN_gl-reconciler-reader", "version": 1 },
      { "type": "agent", "id": "DRYRUN_gl-reconciler-resolver", "version": 1 },
      { "type": "agent", "id": "DRYRUN_gl-reconciler-critic", "version": 1 }
    ],
    "metadata": { "anthropic_cookbook": "<repo-slug>/gl-reconciler" }
  }
]

注意：--dry-run 完全不需要 ANTHROPIC_API_KEY（第 22 行的 :? 檢查被 [[ $DRY_RUN -eq 1 ]] || 短路跳過），但還是需要 jq 和 python3 + pyyaml——沒裝這兩樣，腳本第 40～41 行會直接印 requires jq 或 requires python3 + pyyaml 後結束。

整個 repo 有一支腳本專門把「dry-run 出來的 payload 是不是健康」這件事自動化檢查——scripts/test-cookbooks.sh：對每一個 managed-agent-cookbooks/<slug>/ 跑一次 --dry-run，接著用 Python 檢查三件事：每個 body 的 system 不能是空字串、非最後一筆（也就是 subagent）不能帶 callable_agents（對應「只支援一層委派」的規則）、整份 JSON 裡不能出現 output_schema（它應該在部署時被 del(.output_schema) 拔掉、改用驗證 wrapper）。任何一支 cookbook 沒過，腳本結束碼非 0。

終端機 — 一次跑完全部 cookbook 的 dry-run 健檢
bash scripts/test-cookbooks.sh
#   ✓ gl-reconciler              4 bodies
#   ✓ kyc-screener               4 bodies
#   ...（10 支各印一行）

3.5　日常開發循環：改檔 → sync → check → commit

Skill 的「真本」只放在 plugins/vertical-plugins/<vertical>/skills/；plugins/agent-plugins/<slug>/skills/ 底下的是**同步過去的副本**（給外掛打包用）。日常改動的順序固定是這四步：

1改檔
只改 vertical-plugins/<vertical>/skills/<name>/SKILL.md，不要直接動 agent-plugins/ 底下的副本
→
2sync
python3 scripts/sync-agent-skills.py 把改動複製進每個有掛這個 skill 的 agent 包
→
3check
python3 scripts/check.py 再跑一次，確認沒有漏掉的參照、YAML 沒壞
→
4commit
git commit——pre-commit hook 自動觸發 version_bump.py --apply

scripts/sync-agent-skills.py 的邏輯很直白：先把 vertical-plugins/*/skills/* 全部掃一遍建索引（skill 名稱 → 來源路徑），再對每一個 agent-plugins/*/skills/* 副本找到對應來源，整包 shutil.rmtree + shutil.copytree 覆蓋掉。如果某個副本在 vertical 裡找不到同名來源，會印警告並以結束碼 1 收尾——這通常代表你砍掉了 vertical 裡的 skill 卻忘了同時處理 agent 包裡的殘留副本。

為什麼不能直接改 agent-plugins/ 底下的副本 scripts/check.py 的第 4 項檢查會比對 agent-plugins/<slug>/skills/ 是否跟 vertical-plugins/ 的來源一致，drift 掉會直接擋下 commit（pre-commit hook 跑的就是它）。改錯地方等於改了一份下次 sync 就會被覆蓋掉的副本。

commit 這一步，.githooks/pre-commit（check.py 首次執行時透過 ensure_hooks_installed() 幫你把 git config core.hooksPath 指過去）會呼叫 python3 scripts/version_bump.py --apply：對這個分支上「有 staged 改動、而且版本還沒領先 main」的每個 plugin，把 .claude-plugin/plugin.json 的 version patch 一次（base+1），並且是**冪等**的——同一個 plugin 在分支上 commit 好幾次也只會 bump 一次，不會每次 commit 都加一版。這件事之所以重要，是因為 Claude Code 只有在 plugin 的 version 變了才會重新推送更新給已安裝的使用者；忘記 bump，銀行內部已經裝好這支 agent 的同事就永遠收不到你的修正。PR 上還有 version-bump GitHub Action 當備援，擋掉沒有本機 hook（例如 CI 環境）的貢獻者漏掉這一步。單次 commit 想跳過就用 git commit --no-verify。

3.6　第一天常見卡點

以下是新加入的工程師最常在第一天卡住的地方，逐一攤開（點開看解法）：

缺 jq

deploy-managed-agent.sh 第 40 行會先檢查 command -v jq，找不到就印 requires jq 並結束碼 1。Windows 上用 Git Bash 的話，jq 不會隨 Git 附帶，要另外下載可執行檔放進 PATH；WSL 裡則是 apt install jq 或對應套件管理器裝一下即可。

缺 pyyaml

scripts/check.py 一開頭 import yaml 失敗會印 ERROR: requires pyyaml (pip install pyyaml) 並結束碼 2；deploy-managed-agent.sh 第 41 行也會先探測 python3 -c 'import yaml'，沒有就印 requires python3 + pyyaml。兩支腳本各自獨立檢查，錯誤訊息都直接告訴你要裝什麼——照著 pip install pyyaml 就好。如果機器上有多個 Python（例如系統內建 + pyenv），要確認裝的環境跟 python3 指到的是同一個。

Windows 路徑（反斜線 vs 正斜線）

docs/QuickStart.md 明確提醒：/plugin marketplace add <repo> 要用正斜線 /，例如 c:/Users/.../AnthropicFinancialAgents；反斜線在這個指令下有時會解析失敗。如果路徑格式怎麼調都不行，改用互動模式：直接打 /plugin，切到 Marketplaces 分頁用介面加，對路徑格式比較寬容。另外 scripts/*.sh 系列本來就是 bash script，Windows 原生 cmd/PowerShell 執行不了，要用 Git Bash 或 WSL（見 3.1 節）。

Port 佔用（8001–8009）

mock-mcp/run_all_http.py 固定把 9 支 server 綁在 127.0.0.1:8001 到 8009。如果本機已有其他服務占用同一段 port，該支 server 會啟動失敗——腳本的監控迴圈會偵測到子行程提早退出並印 ! <name> (:<port>) exited with code <code>，但不會自動換 port（沒有這個功能）。這種情況下只能先找出誰佔用了該 port 並關掉，或者改用 --only 只起你需要的那幾支、避開衝突的那個。

marketplace name 撞到官方保留字

本機用 /plugin marketplace add 加這個 repo 時，如果 .claude-plugin/marketplace.json 的 name 還是原本的 claude-for-financial-services，會直接報錯——這個名字是 Anthropic 官方保留的，只能從 anthropics GitHub 組織的來源使用。解法是把本機這份 name 改成別的（docs/QuickStart.md 用 fsi-local 當範例），且**這個改動不要 commit**，純粹是本機安裝用的繞道。

忘記先跑 mock 就跑 dry-run 部署，以為兩者有關聯

容易搞混的一點：--dry-run 只驗證「manifest 能不能正確組成 API payload」，跟 mock MCP server 有沒有在跑**完全無關**——它甚至不需要網路。反過來，MCP server 的 URL 是跑起來後跟 agent 對話時才會真的被打，dry-run 階段看到的 payload 內容不含任何 MCP 呼叫結果。兩件事分別對應「部署設定對不對」和「執行時資料源接不接得到」，不要混為一談去除錯。

ANTHROPIC_API_KEY 沒設就想真部署

deploy-managed-agent.sh 第 22 行只在非 dry-run 時才檢查：[[ $DRY_RUN -eq 1 ]] || : "${ANTHROPIC_API_KEY:?ANTHROPIC_API_KEY must be set}"。忘記設就會在腳本一開頭直接失敗、不會跑到一半才發現。先用 --dry-run 把 payload 確認好，是最省事的除錯順序：先確定「送什麼」，再處理「要不要真的送」。

3.7　本章重點與檢核清單

本章重點 這個 repo 的開發環境刻意做得很薄：沒有建置系統，只有 git／python3+pyyaml／jq／bash 四樣工具。scripts/check.py 是第一個要跑的指令，它同時完成「掛 pre-commit hook」與「lint 全倉庫交叉引用」兩件事。mock-mcp/ 讓 10 支 agent 在沒有任何廠商金鑰、沒有內部系統存取權限的情況下就能離線跑通；deploy-managed-agent.sh --dry-run 讓你在花一次真實 API 呼叫之前，先確認 manifest 真的會被組成合法、depth-1、無 output_schema 外洩的 payload。日常改動永遠走「改 vertical-plugins → sync-agent-skills.py → check.py → commit」這個順序，commit 時 pre-commit hook 會自動幫改到的 plugin patch-bump 版本，讓已安裝的使用者收得到更新。

第一天必完成事項：

  • 裝好 git、python3（含 pyyaml）、jq，Windows 上確認能用 Git Bash 或 WSL 跑 bash 腳本
  • Clone 後跑過一次 python3 scripts/check.py，確認乾淨、且 core.hooksPath 已指到 .githooks
  • 跑過 pip install -r mock-mcp/requirements.txt 並成功用 python3 mock-mcp/run_all_http.py 起過全部 9 支 mock server
  • 跑過 python3 mock-mcp/selftest.py，確認 54 項檢查全過
  • 對至少一支 agent（例如 gl-reconciler）跑過 bash scripts/deploy-managed-agent.sh gl-reconciler --dry-run，看懂輸出的 depth-1 結構
  • 跑過 bash scripts/test-cookbooks.sh，確認 10 支 cookbook 全部 dry-run 健檢通過
  • 知道 skill 真本在哪（vertical-plugins/）、改完要跑 sync-agent-skills.py，不能直接改 agent-plugins/ 副本
Q：你剛剛在 plugins/vertical-plugins/financial-analysis/skills/comps-analysis/SKILL.md 改完同業比較的指標定義，想讓 market-researcher 這支 agent 包也拿到最新版本，接下來該做什麼？
Skill 的真本只在 vertical-plugins/；agent-plugins/<slug>/skills/ 是同步過去的副本。直接改副本會在下次 sync-agent-skills.py 執行時被覆蓋，而且 check.py 的交叉引用檢查會抓到兩邊內容 drift 並擋下 commit。MCP server URL（CAPIQ_MCP_URL 之類）管的是「資料源接哪裡」，跟 skill 文件內容無關；mock server 也只是回傳假資料，不會影響 agent 包裡打包的 skill 檔案。
