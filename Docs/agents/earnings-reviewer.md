# 📊 Earnings Reviewer（財報審閱）

> **用途**：財報追蹤助理，公司發財報後解讀法說會、更新追蹤中的模型，最終產出財報後觀點筆記。

`財報事件 → 讀電話會議＋財報 → 更新追蹤標的的模型 → 草擬財報後筆記 → 分析師審`

| 家族 | 用戶 | 頻率 | 風險 |
|---|---|---|---|
| 研究與建模 | 股票研究分析師 | 每次追蹤的股票發財報 | 🟡 中 |

## 一、Workflow

```
 ticker + 報告期
     │
     ▼
┌────────┐  ┌────────────┐  ┌──────────┐  ┌────────┐  ┌──────────┐
│①拉財報 │─►│②讀電話會議 │─►│③更新模型 │─►│④模型QC │─►│⑤草擬筆記 │
│FactSet │  │earnings-   │  │model-    │  │audit-  │  │morning-  │
│Daloopa │  │analysis    │  │update    │  │xls     │  │note      │
└────────┘  └────────────┘  └──────────┘  └────────┘  └────┬─────┘
                                                            ▼
                                                     資深分析師審 👤
```
> 刻意不做：單一標的的初次納入追蹤（用 model-builder／initiating）

**步驟拆解**
1. **取得財報數字** — 拿到公司剛公布的實際業績、市場原本的預期，以及完整的法說會（財報說明會）內容，不只看摘要。
   - 步驟說明：MCP 資料串接——透過 `factset`／`daloopa` MCP 取得實際業績、市場共識與 10-Q／8-K 文件，並載入完整法說會逐字稿（不靠摘要）。
2. **解讀法說會** — 從會議裡抓出公司對未來的展望、管理層的口風，以及他們刻意避談的問題。
   - 步驟說明：逐字稿解析——以 `earnings-analysis` 技能萃取公司展望、管理層語氣與被迴避的問題，逐字稿僅當不可信資料源處理。
3. **更新模型** — 把新數字更新進原本追蹤這家公司的試算表模型裡。
   - 步驟說明：模型更新——以 `model-update` 技能將新數字寫進活工作簿（持續追蹤的試算表），每個改動的儲存格皆可回溯到資料源。
4. **檢查模型** — 複查模型沒算錯、公式沒斷掉、沒有把數字硬填進去。
   - 步驟說明：模型品質檢查——以 `audit-xls` 技能做平衡驗算、檢查公式連結未斷裂、計算儲存格無硬填數字。
5. **草擬筆記** — 寫一份財報後的觀點筆記，說明這次和預期差在哪、該怎麼解讀。
   - 步驟說明：筆記生成——以 `morning-note` 技能產出筆記框架，填入差異表（實際 vs. 共識 vs. 前估）與對法說會的解讀。
6. **送交審閱** — 把模型和筆記以草稿交出，不直接對外發布。
   - 步驟說明：人工覆核把關——模型與筆記僅以草稿交付，永不對外發布；CMA 版另把寫檔隔離給 note-writer 子代理，分送需資深分析師簽核。

## 二、風險與把關

```
 agent ──草稿──► 資深分析師（內部）      ✗ 永不對外發布

 逐字稿/新聞稿(外來) ──► 只當資料 ──► 流程
 (不可信)
```
> 護操守：每個數字要引用，查不到標 `[UNSOURCED]`　|　最後防線：不發布（分送需分析師簽核）

## 三、技術架構

**技能鏈**
```
earnings-analysis ──► model-update ──► audit-xls ──► morning-note
讀電話會議          更新模型         模型QC        草擬筆記
```
**Skill（共 5 支）**　`earnings-analysis` 讀會議 · `model-update` 更新模型 · `audit-xls` 模型QC · `morning-note` 寫筆記 · `earnings-preview` 財報前瞻
**MCP（2 個）**　`factset` · `daloopa`（皆為公開財務資料供應商，core 外掛已定義）

> 工具：讀＋**寫檔**＋FactSet/Daloopa MCP　|　模型：opus-4-7

**外掛 vs CMA（兩種安裝）**
```
 外掛版 (真人盯)              CMA版 (雲端·無頭)
 ┌──────────────┐            ┌──────────────────────────┐
 │ 單一agent    │            │ 主代理 (不寫)            │
 │ 可直接寫檔   │            │  ├ transcript-reader 讀會議│
 └──────────────┘            │  ├ model-updater   更新   │
  低風險→放寬                │  └ note-writer     唯一可寫│
                            └──────────────────────────┘
```
> 同 meeting-prep：外掛版主代理可直接寫檔（研究草稿、風險中），CMA 版仍把寫檔隔離給 note-writer

**跨 agent**　研究與建模家族 ┄ 與〔market-researcher〕〔model-builder〕互補；初次納入追蹤交給別人

## 五、導入評估

| 面向 | 評估 |
|---|---|
| **導入風險** | 🟡 中 — 產出會影響研究觀點與投資判斷，但僅交草稿、不對外發布、不過帳；資深分析師簽核才發布，可攔錯。風險在數字引用與模型更新正確性，非法規簽核。 |
| **導入成本** | 🟡 中 — `factset`／`daloopa` 公開連接器 core 已定義，只需向供應商買訂閱／API 金鑰；另需客製「持續追蹤標的的活工作簿」模型範本與研究筆記範本，不必自建內部系統。 |
| **適用單位** | 賣方／買方股票研究部、投資團隊 |
| **單位中角色** | 股票研究分析師（下指令＋覆核模型與筆記＋簽核發布）· 資深分析師（最終簽核把關）· 研究助理（草稿與模型加工） |

**最有機會成功的場景**
1. **財報季密集覆蓋的標的更新** — 一個分析師同時追多檔股票，財報集中爆量時最缺人手，自動拉數字＋更新模型＋草擬筆記省最多工，CP 值最高。
2. **法說會（財報說明會）逐字稿快速解讀** — 會議冗長，要在盤後第一時間抓出展望、管理層口風與迴避問題，搶時效產初稿。
3. **模型更新後的品質自查** — 財報後手動改模型最易出公式斷裂或硬填數字，自動跑平衡驗算與連結檢查，降低交付前出錯率。
