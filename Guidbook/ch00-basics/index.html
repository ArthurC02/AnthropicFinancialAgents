一、AI 與 LLM：聊天機器人背後是什麼

你大概用過 Claude 或類似的 AI 聊天工具：打一段字進去，它回一段有模有樣的答案。背後的技術叫 LLM（Large Language Model，大型語言模型）。它的原理說穿了很簡單：讀過海量的文字之後，它非常擅長判斷「接下來最合理的字是什麼」。因為讀得夠多，這種「接話」能力累積成了理解、歸納、翻譯、寫作、甚至推理的本事。

你打給它的那段字，行話叫 提示詞（prompt）。一個好記的比喻：LLM 像一位讀遍全行文件、記性極好的新進同事——你交辦得越清楚，他做得越到位；你只丟一句「幫我看一下這個」，他就只能猜。

但「會聊天」有個天生的限制：它只出一張嘴。你問它「這兩份帳對不對得起來」，它能講方法、能教你步驟，卻不能自己打開檔案、跑比對、把結果寫成報告。要讓 AI 從「顧問」變成「動手做事的人」，就需要下一節的主角。

一句話總結：LLM 是聊天 AI 背後的引擎，很會讀寫與推理，但它本身只能「說」，不能「做」。

二、從「會聊天」到「會做事」：什麼是 agent

agent（智慧代理）＝ LLM ＋ 工具 ＋ 自主的多步驟執行。差別在三件事：

1. 會用工具

不只回答，還能實際查資料庫、讀寫檔案、呼叫其他系統——就像同事不只給建議，還會自己打開系統查帳。

2. 會拆任務

接到「把 3 月 15 日的總帳對一遍」這種大目標，它自己拆成查帳、比對、找差異、寫報告等小步驟，一步步做完。

3. 會停在該停的地方

設計良好的金融 agent 做到「待簽核」就停，把最後的放行留給人——這不是能力不足，而是刻意的安全設計。

用一個對帳的例子看 agent 實際怎麼跑：

1接任務
「核對 3/15 總帳與明細帳」
→
2查資料
用工具向帳務系統調出兩邊的紀錄
→
3找差異
逐筆比對，列出對不起來的項目與可能原因
→
4寫報告
產出差異報告與調整建議
→
5等人簽核
停下來，把「要不要照建議調帳」交給人決定

本書的主角，是 Anthropic 在 financial-services 這個 repo（程式碼儲存庫——一個專案的所有檔案與修改歷史）裡提供的 10 支具名代理（named agent）。「具名」的意思是：每一支都有名字、有明確職掌，像一個個虛擬的專職員工——例如 gl-reconciler 專管總帳對帳、kyc-screener 專管開戶文件初審、meeting-prep-agent 專門幫理專準備客戶會議資料。

agent ≠ 全自動放行這套方案裡風險越高的 agent，人的把關越嚴。對帳、審計這類會碰到「錢」和「法規」的 agent，一律停在「差異報告與建議」，過帳、送出這些動作永遠留給人。第 9、10 章會詳細講這套把關設計。

一句話總結：agent 是「會用工具、會自己拆步驟做完一件事、並在關鍵處停下來等人」的 AI；本書講的是 10 支各有專職的金融 agent。

三、拆開一支金融 agent：四塊積木

把一支 agent 想成一位派駐到你部門的新員工，它的「配備」正好對應四塊積木：

積木比喻白話說明
系統提示詞
（system prompt）		 職位說明書 一份固定的長指示，定義這支 agent 是誰、負責什麼、規矩是什麼、什麼情況要停下來。agent 每次上工都先讀它。
skill（技能） SOP 工作手冊 一份份針對特定任務的操作說明（例如「同業比較怎麼做」「估值模板怎麼填」）。遇到相關任務時，agent 自動翻出對應手冊照著做。
MCP
（Model Context Protocol）		 標準插座／內線總機 agent 連接外部系統（總帳、CRM、行情庫、制裁名單）的標準介面。系統只要提供 MCP 接口，agent 就能即插即用，不必為每套系統另寫程式。
子代理（subagent） 手下的工讀生 主 agent 把小段工作（讀文件、算數字）分派給權限更小的子代理，做完回報。分工同時也是安全隔離——下一段會講為什麼。

子代理的分工有一套固定的形狀：一支 agent 裡有一個 編排者（orchestrator）當主腦，負責接任務、派工、彙整；實際動手的是幾個 葉子工人（leaf worker）。兩條硬規則值得現在就記住，因為全書會反覆出現：

  • 只有一層：編排者可以叫工人，工人不能再往下叫別人。組織圖永遠只有兩層，出事時責任一目了然。
  • 只有一個工人能寫檔：每支 agent 裡，只有一個 leaf worker 有寫入權限。負責讀外來髒資料的工人沒有筆，有筆的工人不碰髒資料——就算文件裡藏了惡意指令（這種攻擊叫提示詞注入），它也拿不到能造成傷害的權限。

對應到 repo 的實際檔案：系統提示詞是 plugins/agent-plugins/<slug>/agents/<slug>.md（每支 agent 一份、全 repo 唯一真本）；skill 是一個個含 SKILL.md 的資料夾，真本放在 plugins/vertical-plugins/<vertical>/skills/，再同步副本進各 agent 包；MCP 連線設定在外掛模式寫在各 vertical 的 .mcp.json、CMA 模式寫在 agent.yaml 的 mcp_servers 欄位；子代理的分工與權限則宣告在 managed-agent-cookbooks/<slug>/subagents/*.yaml。第 2 章會逐一解剖這些檔案。

一句話總結：一支 agent ＝ 職位說明書（系統提示詞）＋ SOP 手冊（skills）＋ 標準插座（MCP）＋ 兩層分工的工讀生（subagents），權限是用結構鎖死的，不是用嘴巴約定的。

四、兩種上工方式：有人陪跑 vs 無人值守

同一支 agent（同一份職位說明書、同一批手冊）可以用兩種方式上工。這是全書最重要的分岔，幾乎每一章都會提到：

外掛（plugin）模式：把 agent 安裝進 Cowork 或 Claude Code 這類有畫面的工具，人坐在螢幕前看著它一步步做，隨時可以插話、修正、喊停。

適合：分析師的日常工作（研究、建模、提案）、試用與評估、開發時的測試。你可以把它想成「新人試用期」——有前輩全程盯著。

CMA（Claude Managed Agents，Claude 託管代理）模式：agent 部署在 Anthropic 雲端，沒有畫面、沒有人盯（行話叫無人值守，headless），由銀行自己的後端程式在事件發生時透過 API（程式對程式的呼叫窗口）叫它起來做事。

適合：夜間批次對帳、月結、開戶文件排隊審查——這些場景沒有人會替每一筆案件手動開視窗。你可以把它想成「轉正式的員工」——照制度自動上工，靠流程與權限把關。

一個常見誤解，先在這裡拆掉本書常說「部署到銀行的 AWS 環境」。要分清楚：agent 本身的運算跑在 Anthropic 雲端；放在銀行 AWS（Amazon 的雲端服務）帳號裡的，是「周邊系統」——呼叫 agent 的後端程式、MCP 連接器、資料來源。第 7 章會畫清楚這條責任邊界。

「一套原始碼、兩種輸出」正是這套方案的核心設計：你不必為兩種模式維護兩套 agent。通常的導入路徑也很自然——先用外掛模式讓業務單位試用、驗證它做得對，再把驗證過的同一支 agent 用 CMA 模式接進正式流程。

一句話總結：外掛模式＝人盯著跑（試用、日常輔助）；CMA＝無人值守、由系統呼叫（批次、正式流程）。同一支 agent，兩種上工方式。

五、先認識這 10 個技術詞，全書無障礙

後面章節難免出現工程詞彙。其實高頻的就這 10 個，現在花三分鐘認識它們，之後看到就不會卡住。（全書任何帶虛線底線的詞都可以點一下看白話解釋；更完整的對照在附錄詞彙表。）

詞一句白話
API程式跟程式對話的窗口；「呼叫 API」＝程式向服務送出請求、拿回結果，全程不用人操作。
JSON / YAML兩種給程式讀的資料格式。JSON 長得像 {"名字": "值"}；YAML 用縮排排版、更好讀，本書的設定檔多用它。
終端機打字下指令的黑視窗。本書所有 python3 ... 開頭的指令都在這裡執行。
部署把做好的東西搬上正式環境開始服務。部署前先在測試環境演練是基本紀律。
mock測試用的假替身：假的帳務系統、假資料，讓你不碰真帳、不用金鑰就能把流程跑通。
環境變數貼在程式執行環境上的「便利貼」：金鑰、網址這類會變動的設定放這裡，不寫死在程式碼裡。
schema資料的格式合約：規定必須有哪些欄位、允許哪些值，不合格式直接退件。
tokenAI 計算文字量的單位（約一個字）。API 按 token 計費——文件越長越貴，跟影印計張數同理。
sessionagent 的一次「上工」：從被叫起來到做完的整段過程與記憶。
人在迴路AI 做、人把關：關鍵動作（送出、過帳）一定停下來等人簽核。全書安全設計的底層原則。

一句話總結：看到不懂的詞，點虛線字或查附錄詞彙表，不用回頭重讀整章。

七、本章重點

本章重點LLM 是「會讀寫推理、但只出一張嘴」的模型；agent ＝ LLM ＋ 工具 ＋ 多步驟執行，而且在關鍵處停下來等人簽核。一支金融 agent 由四塊積木組成：系統提示詞（職位說明書）、skills（SOP 手冊）、MCP（接外部系統的標準插座）、subagents（兩層分工、只有一個工人能寫檔）。同一支 agent 有兩種上工方式：外掛模式有人陪跑、CMA 無人值守跑在 Anthropic 雲端（銀行 AWS 放的是周邊系統）。看不懂的詞，點虛線字即時看解釋。
Q：AI「聊天機器人」跟「agent」最關鍵的差別是什麼？
關鍵差別在「會做事」：agent 能查系統、讀寫檔案、把大任務拆成小步驟一路做完。最後一個選項恰好相反——設計良好的金融 agent 刻意在關鍵決定前停下來等人簽核（人在迴路），風險越高把關越嚴。
